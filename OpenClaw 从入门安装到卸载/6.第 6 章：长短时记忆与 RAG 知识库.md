# 第 6 章：长短时记忆与 RAG 知识库

## 章节前言

普通的 AI 聊天机器人有一个常见问题：一旦换了对话窗口，前面说过的话就很难继续用。你刚解释完业务背景、写作口径或项目约束，下次再聊时往往还得重讲一遍。

OpenClaw 的记忆系统就是为了解决这个问题。它的长期记忆依赖工作区里的记忆文件和索引质量。本章会拆开讲清楚：哪些信息应该写进 `MEMORY.md`，哪些信息应该放进 `memory/YYYY-MM-DD.md`，以及系统是怎么把这些内容重新找回来的。

## 学习目标

读完本章，你将掌握：

*   理解 OpenClaw 短时记忆 -> 压缩 -> 长期记忆的完整流转机制
*   学会配置和维护 `MEMORY.md` 长期记忆文件与每日笔记
*   理解记忆文件、记忆检索与 Agent 工具调用方式
*   实战构建一个可语义搜索的 RAG 个人知识库
*   了解记忆数据的备份、迁移与维护策略

***

## 1. 揭秘 OpenClaw 的双轨记忆机制

大模型的上下文窗口（Context Window）是有限且昂贵的。以 Claude Sonnet 4 为例，虽然它支持 200K tokens 的上下文，但如果每次对话都把几个月前的聊天记录全部塞进去，不仅速度慢、成本高，还会因为"注意力稀释"导致 AI 对关键信息的把握变差。

为了解决这个根本矛盾，OpenClaw 设计了一套**长短时记忆流转机制**，其核心思路是：**用最小的上下文开销，保留最大的信息价值**。

### 1.2 短时记忆：会话状态与自动压缩

短时记忆是你当前正在进行的对话的"工作台"。每当你和 OpenClaw 开始一段新对话，系统会在工作区的 `.openclaw/sessions/` 目录下创建一个独立的会话文件夹，里面保存着：

*   当前对话的完整上下文（你说的话、AI 的回复、工具调用结果）
*   会话元数据（创建时间、关联的渠道、使用的模型等）
*   临时文件和中间产物

随着对话越来越长，上下文会逐渐逼近模型的 token 上限。这时，OpenClaw 会在后台自动触发\*\*压缩（Compaction）\*\*流程：

1.  **信息萃取**：AI 会先扫描即将被压缩的对话片段，提取其中的关键事实、用户偏好、重要决策等高价值信息。
2.  **写入长期存储**：提取出的信息会被分类写入 `MEMORY.md`（稳定信息）或 `memory/YYYY-MM-DD.md`（当日事件）。
3.  **索引更新**：系统会为这些记忆文件建立检索索引，方便后续搜索和回忆。索引由 OpenClaw 自动维护，日常使用时你主要关注记忆文件本身即可。
4.  **安全丢弃**：确认关键信息已经转移后，冗余的对话细节才会被清理，释放上下文空间。

这个过程对用户完全透明——你不会感知到任何中断，AI 也不会突然"失忆"。它就像一个勤劳的秘书，在你不注意的时候默默整理笔记。

### 1.3 长期记忆：工作区记忆文件

长期记忆是 OpenClaw 的"持久存储层"，由两类结构化文件组成：

**`MEMORY.md`——长期稳定信息**

这是存放长期稳定信息的核心文件。你可以手动编辑它，也可以让 AI 在对话中自动维护。**按当前官方口径，`MEMORY.md` 只会在主私聊上下文中加载，不会在群聊上下文中直接带入。** 这样可以降低把个人偏好或私人资料带到群聊里的风险。

以下是一个推荐的 `MEMORY.md` 模板：

```markdown
# 个人记忆库

## 基本信息
- 姓名：张三
- 职业：全栈工程师 / 独立开发者
- 所在城市：深圳
- 常用语言：中文（简体）、英文

## 工作信息
- 公司：XX 科技有限公司
- 主要技术栈：TypeScript, React, Node.js, PostgreSQL
- 当前项目：电商 SaaS 平台重构（预计 2026 年 Q2 完成）
- 团队规模：5 人小团队，我负责前端架构和 DevOps

## 个人偏好
- 写作风格：简洁直接，不喜欢过度修饰的文案
- 代码风格：偏好函数式编程，使用 ESLint + Prettier
- 沟通偏好：重要事项用文字确认，日常沟通可以语音
- 时区：UTC+8（北京时间）
- 工作时间：通常 10:00-22:00，周末偶尔加班

## 常用联系人
- 李四（产品经理）：负责需求对接，飞书沟通
- 王五（后端负责人）：API 接口问题找他
- 赵六（设计师）：UI 相关需求，响应较慢建议提前沟通

## 重要业务知识
- 公司退换货政策：7 天无理由退货，15 天质量问题换货
- 客户分级：VIP 客户（年消费 > 10 万）享受优先客服
- 竞品关注列表：A 公司、B 公司、C 公司

## AI 行为偏好
- 回复长度：中等偏短，避免啰嗦
- 代码生成：始终添加注释，使用 TypeScript 严格模式
- 不确定时：主动询问而非猜测
```

*小贴士：你不需要一次性填完所有内容。可以在日常对话中随时告诉 AI"记住这个"，它会自动更新 `MEMORY.md`。*

**`memory/YYYY-MM-DD.md`——每日笔记**

除了稳定的长期记忆，OpenClaw 还会自动维护每日笔记文件。这些文件按日期命名，记录当天发生的重要事件和上下文。以下是一个典型的每日笔记示例：

```markdown
# 2026-03-14 工作日志

## 今日要事
- 上午与李四对接了 v2.5 版本的需求评审，确认了 3 个新功能点
- 下午修复了支付模块的并发 bug（PR #342 已合并）
- 晚上帮客户 A 公司排查了 API 超时问题，原因是他们的网关配置有误

## 待办跟进
- [ ] 明天需要给王五发送数据库迁移方案的评审邀请
- [ ] 周五前完成 v2.5 的前端原型
- [ ] 联系赵六确认新版 Dashboard 的设计稿进度

## 重要决策
- 决定将缓存方案从 Redis 切换到 Dragonfly，性能提升约 40%
- 与团队达成共识：Q2 暂不支持多租户，优先做好单租户体验

## 学到的新知识
- 发现 Bun 的 SQLite 驱动在高并发写入时需要开启 WAL 模式
- OpenClaw 的记忆索引支持按 Agent 独立管理
```

每日笔记的价值在于：当你第二天继续和 AI 对话时，它能快速回顾昨天的上下文，无缝衔接工作进度，而不需要你重新解释一遍"我们昨天聊到哪了"。

## 2. 记忆检索与 Agent 工具调用

当你的记忆文件越来越多、聊天记录跨度长达几个月时，仅靠人工翻 `MEMORY.md` 和每日笔记会很慢。OpenClaw 会先建立检索索引，再让 Agent 通过内置工具去查找相关内容。

### 2.1 记忆文件与检索机制

OpenClaw 的记忆系统，读者最需要先理解的是这三层：

*   **记忆文件**：`MEMORY.md` 和 `memory/YYYY-MM-DD.md` 默认都放在当前工作区里，都是普通 Markdown 文件，你可以直接查看和编辑。
*   **记忆工具**：Agent 通过 `memory_search` 和 `memory_get` 两个内置工具来检索和读取记忆。
*   **CLI 管理命令**：你可以用 `openclaw memory status`、`openclaw memory index`、`openclaw memory search` 查看状态、重建索引和做搜索测试。

底层索引文件由系统自动维护。你可以知道它存在，但平时不需要把它当作主要操作对象。

记忆的完整工作流程如下：

1.  Agent 在对话过程中，将重要信息写入记忆文件（`MEMORY.md` 或 `memory/日期.md`）
2.  记忆文件的内容被系统建立索引
3.  当需要回忆时，Agent 通过 `memory_search` 工具检索相关记忆
4.  通过 `memory_get` 工具获取特定记忆的完整内容
5.  记忆能否被正确召回，取决于记忆文件的内容质量和索引是否正确建立

> 注意：记忆检索的具体实现（如检索算法、排序策略等）可能随版本变化，请以官方文档为准。

## 3. RAG 实战：把资料整理成可检索知识库

RAG（Retrieval-Augmented Generation，检索增强生成）的意思很简单：**先从资料里找相关内容，再把找到的内容交给模型回答。**

在 OpenClaw 里，你不一定要自己先搭一整套复杂系统。更常见的做法是：把资料交给 Agent，整理成工作区里的 Markdown 记忆文件，再配合检索来回答问题。

### 3.1 实战场景：一键构建可语义搜索的知识库

下面我们用一个完整的实战流程，演示如何把一份 50 页的公司合同变成 AI 可随时调用的知识库。

**第一步：资料投喂（Ingestion）**

你不需要专门去搭复杂的向量数据库。在飞书或 Telegram 聊天框里，直接把文件拖入聊天中：

    你：[拖入文件：公司服务合同_v3.2.pdf]
        请仔细阅读这份合同，提取所有关于退换货政策、
        服务等级协议（SLA）和违约责任的核心条款，
        并将它们存入你的知识库中，以便未来随时调用。

*提示：对于长达 50 页以上的合约或研报，建议将当前会话临时切换至支持超长上下文的模型（如 Gemini 3 Pro 的 1M 上下文或 Kimi K2.5 的 200K 上下文），它们在处理大批量文件整理时表现极佳。你可以在 OpenClaw 的模型设置中切换当前会话使用的模型。*

**第二步：AI 处理与结构化存储**

OpenClaw 收到文件后，会自动执行以下流程：

1.  **文档解析**：调用内置的文档解析器，将 PDF 转换为结构化文本（支持 PDF、Word、Excel、图片 OCR 等格式）。
2.  **智能分块（Chunking）**：系统会先把长文档切成多个小段，方便后面逐段检索和处理。
3.  **信息提取**：根据你的指令，AI 会重点提取退换货政策、SLA、违约责任等关键条款。
4.  **向量化存储**：每个文本片段都会被转换成便于做相似度检索的向量表示。你可以把它理解成“给每段文字生成一个便于搜索的特征值”。
5.  **结构化记录**：提取的核心条款会被整理后写入 `MEMORY.md` 或专门的知识库文件。

AI 的回复可能是这样的：

    OpenClaw：已完成合同分析，提取了以下核心条款并存入知识库：

    📋 退换货政策（共 5 条）：
      1. 标准退货期限：签收后 7 个自然日内
      2. 质量问题换货：签收后 15 个自然日内
      3. 定制商品不支持无理由退货
      ...

    📋 SLA 服务等级（共 3 条）：
      1. 系统可用性承诺：99.9%（月度）
      2. 故障响应时间：P0 级 15 分钟内，P1 级 1 小时内
      ...

    📋 违约责任（共 4 条）：
      ...

    以上内容已存入向量知识库，共生成 127 个记忆片段。
    你可以随时用自然语言查询这些条款。

**第三步：语义召回（Retrieval）**

几天后，客户在群里问了一个刁钻的退货问题：

    客户（在飞书群）：我们买的那批定制款服务器机柜，
        到货后发现尺寸不对，能退吗？

    OpenClaw（自动触发向量搜索，召回合同条款后回复）：
        根据我们的服务合同 v3.2 第 4.3 条规定，
        定制商品原则上不支持无理由退货。但如果存在
        "规格与订单不符"的情况，属于质量问题范畴，
        您可以在签收后 15 个自然日内申请换货。
        建议您提供到货照片和订单规格对比，
        我来协助您发起换货流程。

整个过程中，AI 自动完成了：查询向量数据库 -> 召回相关合同条款 -> 结合上下文生成准确回复。你不需要手动翻合同，也不需要提前告诉 AI "去查一下合同"。

### 3.2 进阶玩法：批量投喂与知识库管理

如果你有大量文档需要投喂，最直接的方式是在对话中逐份或分批发送给 Agent，让它提取关键信息并写入记忆文件。Agent 会自动完成文档解析、信息提取和记忆存储。

对于更大规模的知识库构建，你也可以：

*   手动将整理好的内容写入 `MEMORY.md` 或 `memory/` 目录下的 Markdown 文件
*   让 Agent 在对话中批量处理多份文档，逐一提取并存储关键信息
*   利用脚本将结构化数据转换为 Markdown 格式，放入记忆目录

查看记忆状态：

```bash
# 查看当前记忆索引状态
openclaw memory status

# 手动触发记忆索引重建
openclaw memory index
```

> 提示：记忆文件是纯 Markdown 格式，你可以随时用文本编辑器直接查看和编辑。索引会在 Agent 启动时自动更新。

## 4. 记忆系统的维护建议

随着使用时间的增长，记忆数据会越来越多。以下是一些保持系统高效运行的建议：

### 4.1 记忆文件的组织

良好的记忆文件组织是高质量记忆召回的基础：

*   **`MEMORY.md` 保持精炼**：只存放最核心、最稳定的信息（个人偏好、业务知识、重要决策等）。避免把临时性信息写入主记忆文件。
*   **每日笔记按需记录**：`memory/YYYY-MM-DD.md` 适合记录当天的重要事件和上下文，不需要事无巨细地记录所有对话。
*   **定期整理**：建议每月回顾一次 `MEMORY.md`，删除过时信息，合并重复内容。

### 4.2 索引维护

```bash
# 查看记忆索引状态
openclaw memory status

# 重建记忆索引（当索引与记忆文件不同步时使用）
openclaw memory index
```

### 4.3 性能调优建议

*   **记忆文件的内容质量**：记忆能否被正确召回，很大程度上取决于记忆文件的内容质量。写入记忆时，尽量使用清晰、结构化的描述，避免模糊的表述。
*   **控制记忆文件大小**：单个记忆文件不宜过大。如果 `MEMORY.md` 超过几十 KB，建议拆分为多个主题文件。
*   **索引重建**：如果发现记忆搜索结果不准确，可以尝试重建索引。

## 5. 记忆数据的备份与迁移

你的记忆数据是宝贵的数字资产，务必做好备份。

### 5.1 备份

由于记忆文件都是纯 Markdown 格式，备份非常简单：

```bash
# 备份记忆文件（MEMORY.md 和每日笔记）
cp MEMORY.md ~/backup/
cp -r memory/ ~/backup/memory/

```

> 提示：最重要的是备份工作区里的 Markdown 记忆文件本身。索引通常可以重新生成。

### 5.2 迁移到新设备

将记忆文件复制到新设备的工作区即可：

```bash
# 复制记忆文件到新设备的工作区
cp ~/backup/MEMORY.md /path/to/workspace/
cp -r ~/backup/memory/ /path/to/workspace/memory/

# Agent 启动后会自动重建索引
```

### 5.3 多设备同步

如果你在多台设备上使用 OpenClaw，可以通过以下方式保持记忆同步：

*   **方案一：Git 同步**——将记忆文件纳入 Git 仓库管理，通过 push/pull 同步。适合技术用户。
*   **方案二：云存储同步**——将记忆目录软链接到 iCloud Drive、OneDrive 或坚果云等同步盘目录。适合非技术用户。

## 小结

本章介绍了 OpenClaw 的记忆系统架构：短时记忆负责当前会话的上下文管理，长期记忆通过 `MEMORY.md` 和每日笔记文件持久化重要信息，Agent 再通过 `memory_search` 和 `memory_get` 工具把相关内容找出来。结合 RAG 的思路，你可以把文档整理成可检索的知识库。记忆系统的效果最终还是取决于记忆文件写得是否清楚、是否持续维护。
