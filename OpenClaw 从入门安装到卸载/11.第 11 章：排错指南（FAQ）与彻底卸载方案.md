# 第 11 章：排错指南（FAQ）与彻底卸载方案

## 章节前言

经过前面十章的实战，你的 OpenClaw 现在应该已经配置完善并稳定运行了。但在长期使用中，你难免会遇到网络波动、依赖报错、服务崩溃，甚至想要推倒重来的情况。

本章是全书的排错手册和善后指南。前半部分整理了一套系统化的诊断排错流程和高频 FAQ，帮你在遇到问题时快速定位；后半部分提供了覆盖 macOS、Windows、Linux/Docker 三大平台的完整卸载方案，以及数据备份、迁移和版本回退的操作步骤。最后以全书总结收尾，梳理进阶资源。

## 学习目标

读完本章，你将掌握：

*   熟练使用 `openclaw doctor` 和 `openclaw status` 进行系统诊断
*   能够独立排查并解决 10+ 种常见故障场景
*   掌握日志分析的基本方法，读懂关键日志含义
*   完成 macOS / Windows / Linux / Docker 四种环境的彻底卸载
*   学会数据备份、云端迁移和版本回退的完整操作
*   了解社区资源和进阶学习路径

## 1. 遇到报错不要慌：系统化诊断流程

在使用 OpenClaw 的过程中，如果遇到机器人不回消息、工具调用失败、服务崩溃等情况，不要急着重装。请按照以下三道防线依次排查，绝大多数问题都能在前两步解决。

![配图 11-1：OpenClaw 三道防线排错决策流程图，展示从遇到问题到 openclaw doctor → openclaw status → 网关重启 → 日志分析的完整诊断路径](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/70696a94b0654263844e72715e72e171~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=928\&h=1152\&s=1367551\&e=png\&b=f6f2e2)

### 1.1 第一道防线：先跑 `openclaw doctor`

这是你遇到问题时首先要想到的命令。它会对系统环境、依赖库、配置文件和网络连通性进行一次全面的健康检查，并直接给出修复建议。

在终端执行：

```bash
openclaw doctor
```

**输出类似于：**

> 以下为示意输出，仅供理解 `doctor` 的检查逻辑。实际输出格式和检查项可能随版本变化，请以你本地运行的结果为准。

    # 正常时，输出类似于：
    [PASS] Node.js version: ...
    [PASS] Bun runtime: ...
    [PASS] OS compatibility: ...
    [PASS] Configuration file valid
    [PASS] LLM provider configured
    [PASS] Gateway port 18789 is available
    ...
    Summary: All checks passed.

    # 异常时，输出类似于：
    [FAIL] Bun runtime: not found
           -> Fix: Install Bun via `curl -fsSL https://bun.sh/install | bash`
    [FAIL] API key validation: key expired or invalid
           -> Fix: Update your API key
    [WARN] Disk space low
    ...
    Summary: X FAILED, Y WARNINGS, Z PASSED

`doctor` 会依次检查运行时环境、配置文件、LLM 连通性、端口占用等。如果某项检查失败，输出中会给出对应的修复建议，按提示操作即可。

如果是飞书或 Telegram 连接问题，可以优先参考官方 Channels 文档，并配合 `openclaw channels --help`、`openclaw channels status` 检查当前 channel 状态。

### 1.2 第二道防线：检查 `openclaw status`

如果 `doctor` 检查全部通过但机器人仍然没有反应，你需要确认后台的"心脏"是否还在跳动。

```bash
openclaw status
```

**输出类似于：**

> 以下为示意输出，仅供理解 `status` 的信息结构。实际输出格式可能随版本变化。

    Gateway:
      Status:     RUNNING
      PID:        <进程号>
      Port:       18789
      ...

    Active Sessions:
      （当前活跃的会话列表）

    Scheduled Tasks (Cron):
      （定时任务列表及下次执行时间）

如果 Gateway 状态显示 `STOPPED` 或 `CRASHED`，说明后台服务已经停止，需要进入第三道防线。

### 1.3 第三道防线：必要时重启 Gateway

如果确定是后台服务卡死了，依次执行：

```bash
# 优雅停止
openclaw gateway stop

# 重新启动
openclaw gateway start

# 或者一步到位
openclaw gateway restart
```

如果 `stop` 命令无响应（进程僵死），可以强制终止：

```bash
# 查找并强制终止 Gateway 进程
openclaw gateway stop --force
```

### 1.4 日志分析：深入排查的利器

当三道防线都无法解决问题时，日志是你最后的"黑匣子"。OpenClaw 的日志输出位置取决于你的启动方式和配置，常见情况如下：

*   如果通过 `openclaw gateway start` 前台启动，日志直接输出到终端（stdout/stderr）
*   如果通过 systemd 管理，可以用 `journalctl -u openclaw-gateway` 查看
*   如果通过 Docker 部署，可以用 `docker logs openclaw` 查看（如果你的容器名不是 `openclaw`，请替换为实际名称）
*   具体日志路径请参考你使用版本的官方文档，或运行 `openclaw --help` 查看相关说明

**查看最近的日志（以 systemd 为例）：**

```bash
# 查看最近 100 行日志
journalctl -u openclaw-gateway -n 100

# 实时跟踪日志输出
journalctl -u openclaw-gateway -f

# 按关键词过滤错误日志
journalctl -u openclaw-gateway | grep -i "error\|fatal\|crash"
```

**常见日志含义速查表：**

| 日志关键词                           | 含义                   | 处理方式                      |
| ------------------------------- | -------------------- | ------------------------- |
| `ECONNREFUSED`                  | 网络连接被拒绝（LLM API 不可达） | 检查网络和 API 端点配置            |
| `ENOMEM` / `heap out of memory` | 内存不足，进程被系统杀死         | 增加内存限制或减少并发会话             |
| `ENOSPC`                        | 磁盘空间不足               | 清理日志和临时文件                 |
| `AUTH_FAILED`                   | 认证失败（API Key 无效或过期）  | 更新 API Key                |
| `RATE_LIMIT_EXCEEDED`           | API 调用频率超限           | 降低请求频率或升级 API 套餐          |
| `PLUGIN_CRASH`                  | 插件崩溃                 | 更新或重装对应插件                 |
| `VECTOR_DB_CORRUPT`             | 向量数据库损坏              | 参考 FAQ Q8 进行修复            |
| `CRON_SKIP`                     | 定时任务被跳过              | 检查 Cron 表达式和 Gateway 运行状态 |
| `AGENT_TIMEOUT`                 | 智能体响应超时              | 检查 LLM 延迟，考虑切换模型          |
| `WS_DISCONNECT`                 | WebSocket 连接断开       | 检查网络稳定性，Gateway 会自动重连     |

![配图 11-2：OpenClaw 常见错误日志的分类速查图，按网络类、资源类、认证类、插件类四个维度可视化展示错误关键词与处理方式](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/52f3b1a7b9764325b141612cb3f3ebb8~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1376\&h=768\&s=1398867\&e=png\&b=f6ead4)

## 2. 高频 FAQ：踩坑与填坑记录

以下汇总了社区中最常见的故障场景及解决方案，建议收藏备查。

### Q1：终端提示 `openclaw: command not found` 怎么办？

**原因**：npm 全局安装的目录没有被加到系统的环境变量 `PATH` 中。

**解决方案**：

```bash
# 1. 查看 npm 全局安装路径
npm root -g
# 输出示例：/usr/local/lib/node_modules

# 2. 查看 npm 全局 bin 目录
npm bin -g
# 输出示例：/usr/local/bin

# 3. 将路径添加到 shell 配置文件
echo 'export PATH="/usr/local/bin:$PATH"' >> ~/.zshrc

# 4. 使配置生效
source ~/.zshrc

# 5. 验证
which openclaw
```

如果你使用的是 `nvm` 管理 Node.js 版本，路径可能是 `~/.nvm/versions/node/v22.x.x/bin`，请根据实际情况调整。

### Q2：升级后 Gateway 拒绝启动，报错未授权？

**原因**：当前官方配置要求你显式设置 `gateway.auth.mode`。如果配置里只有 token 或 password，但没有把 `mode` 写清楚，启动和安装流程都可能失败。

**解决方案**：

在 `~/.openclaw/openclaw.json` 中显式添加认证配置：

```json
{
  "gateway": {
    "auth": {
      "mode": "token",
      "token": "your-secure-random-token-here"
    }
  }
}
```

`mode` 当前支持四种值：

*   `"none"`：无认证，仅适合本机临时调试
*   `"token"`：使用固定 Token 认证
*   `"password"`：使用用户名密码认证
*   `"trusted-proxy"`：信任反向代理转发的身份头

配置完成后重启 Gateway：`openclaw gateway restart`。

### Q3：AI 突然变成哑巴，只会聊天但拒绝执行任何工具和命令？

**原因**：2026.3.2 版本更新中，工具权限做了强隔离，默认 Profile 被改为纯聊天模式（`messaging`）。

**解决方案**：

修改 `~/.openclaw/openclaw.json`，将工具配置改为完整模式：

```json
{
  "tools": {
    "profile": "full"
  }
}
```

或者通过 Web 控制台进入 Settings -> Agent -> Tools Profile，选择 `Full`，然后重启网关。

### Q4：内存不足导致 Gateway 崩溃（OOM Killed）？

**症状**：Gateway 运行一段时间后突然退出，日志中出现 `ENOMEM`、`heap out of memory` 或系统日志显示 `OOM Killed`。

**原因**：长时间运行的对话积累了大量上下文，或并发会话过多导致内存溢出。在 2 核 4G 的云服务器上尤为常见。

**解决方案**：

```bash
# 1. 增加 Node.js 堆内存上限（默认约 1.5GB）
export NODE_OPTIONS="--max-old-space-size=3072"

# 2. 写入配置文件使其永久生效
echo 'export NODE_OPTIONS="--max-old-space-size=3072"' >> ~/.zshrc
source ~/.zshrc

# 3. 重启 Gateway
openclaw gateway restart
```

如果是 Docker 部署，还需要限制容器内存：

```bash
# 限制容器最大使用 3GB 内存，预留 1GB 给宿主机
docker update --memory=3g --memory-swap=4g openclaw-container
```

长期方案：

*   启用向量记忆（第 6 章），减少每次对话的上下文长度
*   定期通过 Web 控制台清理不活跃的会话
*   升级服务器配置到 4 核 8G 以上

### Q5：Skills 安装失败，提示网络超时或校验错误？

**症状**：执行 `clawhub install xxx` 时报错 `ETIMEOUT`、`CHECKSUM_MISMATCH` 或 `REGISTRY_UNREACHABLE`。

**解决方案**：

```bash
# 1. 检查网络连通性（确认能访问 ClawHub 注册表）
curl -I https://clawhub.ai

# 2. 如果是网络问题，检查代理配置或尝试切换网络环境

# 3. 清除本地缓存后重试安装
# 具体缓存清理命令请参考 clawhub --help 的输出

# 4. 重新尝试安装
clawhub install xxx
```

安装第三方 Skill 前，务必在 ClawHub 上查看其来源和评价，确认安全性（参考第 10 章安全红线）。

### Q6：飞书 / Telegram 连接频繁断开？

**症状**：IM 插件隔一段时间就断线，需要手动重连。日志中出现 `WS_DISCONNECT` 或 `WEBHOOK_TIMEOUT`。

**原因**：

*   飞书：连接模式配置不匹配，或 Webhook 回调地址失效
*   Telegram：网络不稳定导致 Long Polling 超时

**解决方案**：

**飞书：**

```bash
# 1. 查看当前 channel 状态
openclaw channels status

# 2. 如需排查详细日志
openclaw channels logs
```

然后检查 `channels.feishu.connectionMode` 是否与你的部署方式一致：

*   本地或内网环境通常优先用 `websocket`
*   `webhook` 模式需要可访问的回调地址，以及飞书要求的 `verificationToken` 和 `encryptKey`

**Telegram：**

```bash
# 1. 查看当前 channel 状态
openclaw channels status

# 2. 如需排查详细日志
openclaw channels logs
```

Telegram 当前默认就是 Long Polling。只有在你已经具备稳定公网 HTTPS 回调地址时，才建议改用 Webhook；否则先检查 `botToken`、网络连通性和代理设置。

### Q7：定时任务（Cron Job）不执行？

**症状**：配置了定时任务但到了时间没有触发，`openclaw status` 中 Cron 状态显示 `skipped` 或 `missed`。

**原因**：

*   Gateway 在任务触发时间点未运行
*   Cron 表达式语法错误
*   时区配置不一致

**解决方案**：

```bash
# 1. 检查所有定时任务状态
openclaw cron list

# 2. 手动触发一次，验证任务本身是否正常
openclaw cron run cron_01

# 3. 检查时区配置
openclaw config get timezone
# 如果为空，设置为你的本地时区
openclaw config set timezone "Asia/Shanghai"

# 4. 验证 Cron 表达式（推荐使用 crontab.guru 在线工具）
# 例如 "0 9 * * *" 表示每天上午 9 点
```

确保 Gateway 设置了开机自启（参考第 2 章），否则服务器重启后定时任务会全部失效。

### Q8：向量数据库损坏，记忆搜索失败？

**症状**：AI 突然"失忆"，无法检索历史对话。日志中出现 `VECTOR_DB_CORRUPT` 或 `SQLITE_CORRUPT`。

**原因**：通常是非正常关机（断电、强制 kill 进程）导致 SQLite 数据库文件损坏。

**解决方案**：

```bash
# 1. 检查 SQLite 数据库完整性
sqlite3 ~/.openclaw/data/vector.db "PRAGMA integrity_check;"

# 2. 如果数据库损坏，从备份恢复
cp ~/.openclaw/backups/vector.db.bak ~/.openclaw/data/vector.db

# 3. 如果没有备份，可能需要删除损坏的数据库文件并重启 Gateway
#    重启后系统会创建新的空数据库（历史向量数据会丢失）
#    具体的记忆数据修复命令请参考当前版本的 openclaw memory --help
```

强烈建议配置自动备份（参考本章第 4 节），避免数据丢失。

### Q9：多智能体通信失败，Agent 之间无法协作？

**症状**：配置了多个 Agent（如技术助理和生活助理），但它们之间无法传递消息或共享上下文。

**原因**：

*   Agent 路由配置错误，消息被发送到了错误的 Agent
*   工作空间隔离过于严格，阻断了跨 Agent 通信
*   Gateway 的 Agent 注册表未更新

**解决方案**：

```bash
# 1. 查看当前 Agent 列表
openclaw agents list

# 2. 查看当前绑定关系
openclaw agents bindings
```

然后直接发一条真实测试消息，确认消息是否被路由到预期的 Agent。比起依赖不存在的旧命令，实际发消息验证更可靠。

### Q10：API Key 配额耗尽，AI 完全停止响应？

**症状**：所有对话请求返回 `429 Too Many Requests` 或 `QUOTA_EXCEEDED`，AI 完全无法使用。

**解决方案**：

```bash
# 1. 登录你的 LLM Provider 控制台查看当前 API 用量

# 2. 临时切换到备用模型（如果配置了多个 LLM Provider）
#    直接修改 ~/.openclaw/openclaw.json 中的 models.providers.* 和 agents.defaults.model
openclaw gateway restart

# 3. 配置多 Provider 自动降级（推荐长期方案）
```

在 `openclaw.json` 中配置多个 Provider：

```json
{
  "models": {
    "providers": {
      "deepseek": {
        "baseUrl": "https://api.deepseek.com/v1",
        "apiKey": "sk-your-deepseek-key"
      },
      "zhipu": {
        "baseUrl": "https://open.bigmodel.cn/api/paas/v4",
        "apiKey": "your-zhipu-key"
      }
    }
  },
  "agents": {
    "defaults": {
      "model": "<你的主模型ID>"   // 主模型不可用时，手动切换为备用模型 ID
    }
  }
}
```

这样当主 Provider 出现问题时，你可以快速切换到备用 Provider，保证服务不中断。

## 3. 完美告别：全平台深度清理与干净卸载指南

有时候，由于配置被改得太乱，或者你想要迁移服务器，最好的办法就是彻底卸载重装。但很多新手发现，仅仅卸载了 npm 包后，原来的对话记录和后台进程依然存在。以下提供覆盖三大平台的"拔草除根"级深度卸载方案。

### 3.1 macOS 深度卸载

![配图 11-4：三大平台（macOS / Windows WSL2 / Linux Docker）卸载步骤对照表，帮助读者快速定位自己平台的卸载流程](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/383384410feb4dc9925ecc1b1c0c8be8~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1376\&h=768\&s=1276539\&e=png\&b=f6f3e9)

**第一步：使用内置卸载程序（推荐）**

```bash
openclaw uninstall
```

系统会弹出交互式选项，建议将以下四项全部勾选：

*   Gateway service（后台服务）
*   State（状态数据）
*   Workspace（工作区）
*   macOS app（应用本体）

这能释放已申请的系统权限并注销官方服务。

**第二步：卸载 CLI 核心程序**

根据你的安装方式执行卸载：

```bash
# npm 安装方式
npm rm -g openclaw

# Homebrew 安装方式
brew uninstall openclaw
brew autoremove  # 清理无用依赖
```

验证卸载结果：

```bash
which openclaw
# 应输出：openclaw not found
```

**第三步：深度清理工作区与记忆残留**

OpenClaw 默认会在用户目录下创建配置文件、数据库和日志。如果你不再需要任何历史数据，执行以下命令将其物理删除：

```bash
# 删除所有 OpenClaw 数据（不可逆操作！）
rm -rf ~/.openclaw
```

> 注意：执行此命令前请务必确认你不需要保留原来的数据，或者已经完成了备份（参考第 4 节）。

**第四步：猎杀僵尸守护进程（LaunchAgent）**

如果你曾经把 OpenClaw 设置为开机自启，它可能还在后台运行。清理 macOS 的 plist 启动配置文件：

```bash
# 停止并卸载 LaunchAgent
launchctl bootout gui/$(id -u) ~/Library/LaunchAgents/ai.openclaw.gateway.plist 2>/dev/null

# 删除 plist 文件
rm -f ~/Library/LaunchAgents/ai.openclaw.gateway.plist

# 确认没有残留进程
ps aux | grep -i openclaw | grep -v grep
# 如果仍有进程，手动终止
# kill -9 <PID>
```

### 3.2 Windows（WSL2）深度卸载

如果你按照第 2 章的指引在 WSL2 环境中安装了 OpenClaw，卸载需要同时清理 WSL 内部和 Windows 宿主机两侧。

**第一步：在 WSL2 内部卸载**

打开 WSL2 终端（Ubuntu）：

```bash
# 停止 Gateway
openclaw gateway stop

# 使用内置卸载程序
openclaw uninstall

# 卸载 npm 全局包
npm rm -g openclaw

# 清理工作区数据
rm -rf ~/.openclaw

# 如果配置了 systemd 自启服务
sudo systemctl stop openclaw-gateway
sudo systemctl disable openclaw-gateway
sudo rm -f /etc/systemd/system/openclaw-gateway.service
sudo systemctl daemon-reload
```

**第二步：清理 Windows 宿主机侧**

在 PowerShell（管理员模式）中执行：

```powershell
# 如果在 Windows 侧创建了快捷方式或计划任务
schtasks /delete /tn "OpenClawGateway" /f 2>$null

# 清理 Windows 侧的残留配置（如果有）
Remove-Item -Recurse -Force "$env:USERPROFILE\.openclaw" -ErrorAction SilentlyContinue
```

**第三步：（可选）彻底移除 WSL2 发行版**

如果这个 WSL2 发行版是专门为 OpenClaw 创建的，可以直接销毁：

```powershell
# 查看已安装的 WSL 发行版
wsl --list --verbose

# 注销指定发行版（会删除该发行版的所有数据！）
wsl --unregister Ubuntu
```

### 3.3 Linux / Docker 深度卸载

**原生 Linux 安装卸载：**

```bash
# 停止服务
openclaw gateway stop

# 使用内置卸载程序
openclaw uninstall

# 卸载 npm 全局包
npm rm -g openclaw

# 清理数据目录
rm -rf ~/.openclaw

# 如果配置了 systemd 服务
sudo systemctl stop openclaw-gateway
sudo systemctl disable openclaw-gateway
sudo rm -f /etc/systemd/system/openclaw-gateway.service
sudo systemctl daemon-reload

# 清理日志
sudo rm -rf /var/log/openclaw/
```

**Docker 部署卸载：**

```bash
# 停止并删除容器
docker stop openclaw
docker rm openclaw

# 删除镜像
docker rmi ghcr.io/openclaw/openclaw:latest

# 如果你使用的是命名卷，再删除对应卷（会一并删掉其中的数据）
# docker volume rm openclaw_data

# 如果使用了 docker-compose

```

如果你使用了独立的数据目录挂载：

```bash
# 删除挂载的数据目录
rm -rf /opt/openclaw-data
```

## 4. 数据备份与云端迁移

在卸载或迁移之前，强烈建议先做好数据备份。OpenClaw 的核心数据都集中在 `~/.openclaw` 目录下。

### 4.1 手动备份

```bash
# 创建带时间戳的完整备份
BACKUP_NAME="openclaw-backup-$(date +%Y%m%d-%H%M%S)"
tar -czf ~/$BACKUP_NAME.tar.gz -C ~ .openclaw

echo "Backup created: ~/$BACKUP_NAME.tar.gz"
echo "Size: $(du -sh ~/$BACKUP_NAME.tar.gz | cut -f1)"
```

如果只需要备份关键数据（不含日志和缓存）：

```bash
# 精简备份（仅配置 + 数据库 + 记忆）
BACKUP_NAME="openclaw-essential-$(date +%Y%m%d-%H%M%S)"
mkdir -p /tmp/$BACKUP_NAME

cp ~/.openclaw/openclaw.json /tmp/$BACKUP_NAME/
cp -r ~/.openclaw/data/ /tmp/$BACKUP_NAME/data/
cp -r ~/.openclaw/agents/ /tmp/$BACKUP_NAME/agents/
cp -r ~/.openclaw/memory/ /tmp/$BACKUP_NAME/memory/

tar -czf ~/$BACKUP_NAME.tar.gz -C /tmp $BACKUP_NAME
rm -rf /tmp/$BACKUP_NAME

echo "Essential backup created: ~/$BACKUP_NAME.tar.gz"
```

### 4.2 自动定时备份脚本

创建一个自动备份脚本，配合 Cron 实现每日自动备份：

```bash
#!/bin/bash
# ~/.openclaw/scripts/auto-backup.sh

BACKUP_DIR="$HOME/.openclaw/backups"
MAX_BACKUPS=7  # 保留最近 7 天的备份

mkdir -p "$BACKUP_DIR"

# 创建备份
BACKUP_FILE="$BACKUP_DIR/openclaw-$(date +%Y%m%d).tar.gz"
tar -czf "$BACKUP_FILE" \
  -C "$HOME" \
  .openclaw/openclaw.json \
  .openclaw/data/ \
  .openclaw/agents/ \
  .openclaw/memory/ \
  2>/dev/null

# 清理过期备份（保留最近 N 个）
ls -t "$BACKUP_DIR"/openclaw-*.tar.gz 2>/dev/null | tail -n +$((MAX_BACKUPS + 1)) | xargs rm -f

echo "[$(date)] Backup completed: $BACKUP_FILE"
```

```bash
# 赋予执行权限
chmod +x ~/.openclaw/scripts/auto-backup.sh

# 添加到 crontab（每天凌晨 3 点执行）
(crontab -l 2>/dev/null; echo "0 3 * * * ~/.openclaw/scripts/auto-backup.sh >> ~/.openclaw/logs/backup.log 2>&1") | crontab -
```

### 4.3 从本地迁移到云端服务器

当你决定将 OpenClaw 从本地 Mac 迁移到云服务器（如腾讯云 Lighthouse）实现 24 小时运行时，按以下步骤操作：

**第一步：在本地创建备份并上传**

```bash
# 本地：创建备份
tar -czf ~/openclaw-migration.tar.gz -C ~ .openclaw

# 上传到云服务器（替换为你的服务器 IP）
scp ~/openclaw-migration.tar.gz root@your-server-ip:/root/
```

**第二步：在云服务器上恢复**

```bash
# 云服务器：安装 OpenClaw（确保 Node.js >= v22.16，推荐 Node 24）
npm install -g openclaw

# 恢复数据
tar -xzf /root/openclaw-migration.tar.gz -C /root/

# 修改配置（云端通常仍建议保持 loopback + 反向代理）
# 如需对外提供服务，优先保留 gateway.bind="loopback"，
# 再通过反向代理 + 认证对外暴露
```

**第三步：验证并启动**

```bash
# 运行健康检查
openclaw doctor

# 启动 Gateway
openclaw gateway start

# 确认状态
openclaw status

# 配置开机自启（参考第 2 章中 systemd 或 LaunchAgent 的配置方式）
```

> 安全提醒：迁移到云端后，务必参考第 10 章配置 Nginx 反向代理和 Gateway 认证，不要将服务裸露在公网。

## 5. 版本回退：当升级翻车时的后悔药

有时候新版本可能引入了 Breaking Change 或 Bug，导致你的工作流崩溃。这时候你需要回退到上一个稳定版本。

### 5.1 npm 安装方式回退

> 以下版本号仅作为截至 2026-03-14 的示例，实际可回退版本请以 `npm view openclaw versions --json` 的结果为准。

```bash
# 1. 查看当前版本
openclaw --version
# 输出：2026.3.7

# 2. 查看所有可用版本
npm view openclaw versions --json | tail -20

# 3. 回退到指定版本
npm install -g openclaw@2026.3.5

# 4. 验证回退结果
openclaw --version
# 输出：2026.3.5

# 5. 重启 Gateway
openclaw gateway restart

# 6. 运行健康检查确认一切正常
openclaw doctor
```

如果你想锁定版本，防止意外升级：

```bash
# 查看 npm 全局包的更新策略
npm config set save-exact true
```

### 5.2 Docker 部署方式回退

> 下面的镜像标签同样是截至 2026-03-14 的示例，请按你实际需要回退到对应版本。

```bash
# 1. 查看当前运行的镜像版本
docker inspect openclaw --format='{{.Config.Image}}'

# 2. 停止当前容器
docker stop openclaw
docker rm openclaw

# 3. 拉取旧版本镜像
docker pull ghcr.io/openclaw/openclaw:2026.3.5

# 4. 使用旧版本重新启动（保持原有的数据卷挂载）
#    这里请补回你原来使用的 -v 挂载参数，不要临时改数据路径
docker run -d \
  --name openclaw \
  --restart unless-stopped \
  -p 18789:18789 \
  ghcr.io/openclaw/openclaw:2026.3.5

# 5. 验证
docker logs openclaw --tail 20
```

如果使用 `docker-compose`，修改 `docker-compose.yml` 中的镜像版本号：

```yaml
services:
  openclaw:
    image: ghcr.io/openclaw/openclaw:2026.3.5  # 修改为目标版本
    # ... 其他配置保持不变
```

然后执行：

```bash
docker compose up -d
```

### 5.3 回退后的注意事项

*   **配置兼容性**：新版本可能引入新的配置字段。回退后如果启动报错，先对照回退版本的官方文档检查 `gateway.auth.mode`、模型配置和 channels 配置。
*   **数据库兼容性**：向量数据库的 Schema 可能在版本间有变化。如果回退后出现数据库错误，可能需要参考当前版本的 `openclaw memory --help` 进行索引重建。
*   **插件兼容性**：某些插件可能依赖特定版本的 OpenClaw Core API。回退后如果插件报错，尝试更新或重新安装相关插件；插件命令请以 `openclaw plugins --help` 为准。
