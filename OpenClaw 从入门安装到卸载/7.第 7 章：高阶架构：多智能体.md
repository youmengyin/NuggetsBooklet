# 第 7 章：高阶架构：多智能体（Multi-Agent）与自动化编排

## 章节前言

在前面的章节中，我们已经让 OpenClaw 具备了工具调用和持久记忆能力。但单个 Agent 的能力终归有限——当你需要不同渠道使用不同的 AI 角色，或者让多个 Agent 并行处理不同类型的任务时，就需要多智能体架构了。

本章将带你进入 OpenClaw 的高阶玩法：通过**多智能体配置（Multi-Agent）**和**自动化引擎**，让你的 AI 从"被动答疑"转变为"主动工作"——7x24 小时不间断地巡检、汇报、执行任务。

## 学习目标

读完本章，你将掌握：

*   理解多智能体架构的核心概念与配置方式
*   学会在 `openclaw.json` 中通过 `agents.list[]` 和 `bindings[]` 配置多个独立 Agent
*   掌握 Session Tools（`sessions_list`、`sessions_send`、`sessions_spawn`）作为工具调用的完整用法
*   理解 Heartbeat 心跳机制的工作原理
*   通过 CLI 管理 Cron Job 定时任务
*   了解 Webhooks / Hooks 外部事件触发的接入思路
*   掌握 per-agent 的沙箱隔离配置（mode 和 scope）
*   了解多智能体调试与监控的实用方法

## 1. 组建虚拟团队：多智能体路由（Multi-Agent Routing）

默认情况下，你所有的聊天渠道（飞书、Telegram）都在和同一个 Agent 对话。但这会带来权限和设定的冲突——比如，你希望对外的客服机器人严谨且只能查资料，而你私用的助理可以直接执行服务器命令。

OpenClaw 支持通过 `agents.list[]` 配置多个独立 Agent，并通过 `bindings[]` 将不同的聊天渠道绑定到指定的 Agent，实现人设、模型和权限的完全隔离。

### 1.2 完整的 `openclaw.json` 多智能体配置

多智能体配置的核心在于 `openclaw.json` 中的 `agents` 和 `bindings` 两个顶层字段。以下是一个包含 4 个 Agent 的完整配置示例：

```json
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "anthropic/your-model-id"
      }
    },
    "list": [
      {
        "id": "personal-assistant",
        "name": "私人助理小龙虾",
        "description": "私人全能助理，拥有最高权限",
        "model": {
          "primary": "anthropic/your-model-id"
        }
      },
      {
        "id": "tech-expert",
        "name": "技术专家",
        "description": "专注代码审查、技术方案和 DevOps",
        "model": {
          "primary": "anthropic/your-model-id"
        }
      },
      {
        "id": "strategy-advisor",
        "name": "战略顾问",
        "description": "商业分析、市场调研和战略规划",
        "model": {
          "primary": "openai/your-model-id"
        }
      },
      {
        "id": "community-mod",
        "name": "社区版主",
        "description": "Discord/Telegram 社区管理和用户答疑",
        "model": {
          "primary": "deepseek/deepseek-chat"
        }
      }
    ]
  },
  "bindings": [
    {
      "channel": "feishu",
      "agent": "personal-assistant"
    },
    {
      "channel": "telegram",
      "agent": "community-mod"
    },
    {
      "channel": "discord",
      "agent": "community-mod"
    }
  ]
}
```

配置要点解读：

*   **`agents.defaults`**：为所有 Agent 设置默认值（如默认模型），单个 Agent 可以覆盖默认值。
*   **`agents.list[]`**：以数组形式定义所有可用的智能体，每个 Agent 通过 `id` 唯一标识，拥有独立的名称、描述和模型配置。
*   **`bindings[]`**：定义渠道（channel）到 Agent 的绑定关系。每条 binding 指定一个 `channel` 和一个 `agent`，将特定渠道的消息路由到对应的 Agent 处理。
*   **模型写法**：当前更稳妥的写法是统一用 `provider/model`。例如 `openai/your-model-id`、`deepseek/deepseek-chat`、`anthropic/your-model-id`。

### 1.3 Agent 独立工作空间的目录结构

下面这个目录结构是**示意写法**，目的是帮助你理解“不同 Agent 可以使用不同工作区和提示文件”。它不是所有部署都必须手工照着创建的一套固定目录：

    ~/.openclaw/
    ├── openclaw.json              # 全局配置（包含多智能体路由）
    ├── workspaces/
    │   ├── personal/              # 私人助理工作区

    │   │   ├── SOUL.md            # 性格与语气设定
    │   │   ├── MEMORY.md          # 长期记忆（个人偏好、联系人等）
    │   │   ├── memory/            # 每日记忆笔记
    │   │   │   ├── 2026-03-12.md
    │   │   │   └── 2026-03-13.md
    │   │   └── sessions/          # 会话历史
    │   ├── tech/                  # 技术专家工作区
    │   │   ├── AGENTS.md          # 技术领域专精设定
    │   │   ├── SOUL.md            # 严谨、精确的语气
    │   │   ├── MEMORY.md          # 项目技术栈、架构决策记录
    │   │   └── memory/
    │   ├── strategy/              # 战略顾问工作区
    │   │   ├── AGENTS.md          # 商业分析方法论
    │   │   ├── SOUL.md            # 商务正式语气
    │   │   ├── MEMORY.md          # 行业数据、竞品信息
    │   │   └── memory/
    │   └── community/             # 社区版主工作区
    │       ├── AGENTS.md          # 社区规则、FAQ 知识库
    │       ├── SOUL.md            # 友好、耐心的语气
    │       ├── MEMORY.md          # 常见问题、活跃用户记录

这种隔离设计的好处是显而易见的：

*   **记忆不串台**：技术专家不会突然聊起你的私人日程，社区版主也不会泄露公司内部的技术方案。
*   **人设不冲突**：每个 Agent 都有自己的"性格"，不会因为上一个对话的语境而"人格分裂"。
*   **权限最小化**：遵循安全最佳实践，每个 Agent 只能访问自己工作区内的文件和被授权的工具。

### 1.4 智能体间通信：Session Tools 详解

多个 Agent 各自为战是不够的，真正的团队需要协作。OpenClaw 内置了一组**会话工具（Session Tools）**，允许不同频道的 Agent 互相传话、委派任务和汇报结果。

#### `sessions_list`：查看活跃会话

`sessions_list` 是 Agent 可以调用的一个内置工具（tool call），用于获取当前系统中所有活跃会话的列表。它不是 shell 命令，而是 Agent 在对话过程中自主调用的工具。

当你对 Agent 说"帮我看看现在有哪些 Agent 在线"时，Agent 会发起一次 `sessions_list` 工具调用：

```json
{
  "tool": "sessions_list",
  "parameters": {}
}
```

返回结果是一个会话数组，包含每个活跃会话的 ID、关联 Agent、渠道和状态信息。Agent 会将这些信息整理后以自然语言回复你。

#### `sessions_send`：跨会话发送消息

`sessions_send` 同样是 Agent 的内置工具调用。当一个 Agent 需要向另一个会话中的 Agent 发送消息时，会调用此工具。这里的目标是 **session key**，不是群名，也不是 Agent 名称。

例如，你在飞书私聊中说"帮我通知技术讨论群，今晚 8 点有代码评审会议"，私人助理会发起如下工具调用：

```json
{
  "tool": "sessions_send",
  "parameters": {
    "target_session": "sess_d4e5f6",
    "message": "通知：今晚 20:00 有代码评审会议，请相关同学准时参加。"
  }
}
```

消息会被投递到目标会话，由对应的 Agent 在其绑定的渠道中发出。所以在实际使用里，前一步通常要先通过 `sessions_list` 找到正确的 session key。

#### `sessions_spawn`：创建子会话委派任务

这是多智能体协作中最核心的工具调用。一个 Agent 可以通过 `sessions_spawn` 创建一个新的子会话，将任务委派给另一个 Agent，等对方完成后接收结果。

例如，你说"帮我调研一下竞品 A 公司最近的产品动态，然后写一份分析报告"，私人助理判断这属于市场调研任务，会发起如下工具调用：

```json
{
  "tool": "sessions_spawn",
  "parameters": {
    "agent": "strategy-advisor",
    "task": "调研竞品 A 公司最近 30 天的产品动态，包括新功能发布、定价变化、用户反馈，输出一份结构化分析报告。",
    "callback": true
  }
}
```

`sessions_spawn` 执行后，系统会创建一个子会话，战略顾问在后台独立工作（调用浏览器、搜索等工具），完成后将结果回调给发起者。私人助理收到结果后，整理并回复你。

`sessions_spawn` 的关键特性：

*   **异步执行**：委派任务后，主 Agent 可以继续处理其他事务，不会被阻塞。
*   **结果回调**：设置 `callback: true` 后，子会话完成时自动将结果返回给发起者。
*   **链式委派**：子会话中的 Agent 还可以继续调用 `sessions_spawn`，形成多层级的任务分解。

### 1.5 多智能体协作实战：一个完整的对话流程

让我们通过一个真实场景，看看多个 Agent 是如何协同工作的：

    场景：你在飞书私聊中说"帮我准备明天的产品评审会议材料"

    [Step 1] 私人助理（飞书私聊）接收指令
      ├── 分析任务：需要技术方案 + 市场数据 + 会议议程
      ├── 工具调用 sessions_spawn(agent: "tech-expert", task: "整理当前项目的技术进展、未解决的 Bug 列表和性能指标")
      ├── 工具调用 sessions_spawn(agent: "strategy-advisor", task: "收集本周竞品动态和用户反馈数据")
      └── 同时开始起草会议议程模板

    [Step 2] 技术专家（后台子会话）独立工作
      ├── 工具调用 shell：git log --since="1 week ago"
      ├── 工具调用 shell：查询 Bug 追踪系统 API
      ├── 整理技术进展报告
      └── 完成，结果回调给私人助理

    [Step 3] 战略顾问（后台子会话）独立工作
      ├── 工具调用 browser：访问竞品官网
      ├── 工具调用 web-search：搜索行业新闻
      ├── 整理市场分析报告
      └── 完成，结果回调给私人助理

    [Step 4] 私人助理汇总所有结果
      ├── 合并技术报告 + 市场分析 + 会议议程
      ├── 生成完整的会议材料文档
      └── 推送给你："会议材料已准备完毕，共 3 个部分..."

整个过程中，你只需要一句话，三个 Agent 就会通过工具调用自动分工、并行执行、汇总交付。

## 2. 自动化引擎：从"被动响应"到"主动出击"

传统的 AI 是你问它才答，而 OpenClaw 还支持几种“主动触发”的方式。这里先把边界说清楚：**Cron 和 Heartbeat 属于 Gateway 内建能力；Webhooks/Hooks 往往还需要外部系统或中间件配合。**

### 2.1 引擎 A：Heartbeat（心跳机制）

**Heartbeat** 是 OpenClaw 的周期性唤醒机制。系统会按照设定的间隔定期唤醒 Agent，让它检查是否有需要主动处理的事项，并在必要时向你发送消息。

**工作原理：**

1.  Gateway 按照设定的间隔向 Agent 发送心跳信号
2.  Agent 被唤醒后，检查待办事项、监控指标、未读消息等
3.  如果发现需要处理的事项，Agent 主动发起对话或执行操作
4.  处理完毕后，Agent 回到休眠状态，等待下一次心跳

**配置方式：**

Heartbeat 的具体配置取决于你的部署方式和版本。核心概念是为指定的 Agent 开启心跳并设置触发间隔。典型的配置项包括：

*   **触发间隔**：心跳的周期，例如每 5 分钟唤醒一次
*   **检查内容**：Agent 被唤醒后需要检查的事项类型（待办任务、系统告警、定时提醒等）
*   **免打扰时段**：设置静默时间窗口，非紧急消息会被缓存到免打扰结束后统一推送

**实际效果：** 很多社区用户反馈，当他们在忙碌时，OpenClaw 会主动发消息说"我发现某个测试跑完了，结果如下"或者"你的服务器内存使用率达到 92%，我已经自动清理了临时文件"。这种主动汇报的体验，是传统 AI 无法提供的。

> **小贴士：** 免打扰时段的配置非常重要。没有人希望凌晨 3 点被 AI 的主动汇报吵醒。设置免打扰时段后，心跳检查仍会执行，但非紧急消息会被缓存到免打扰结束后统一推送。

### 2.2 引擎 B：Cron Job（定时任务）

**Cron Job** 用于在精确的时间点执行预设任务。与 Heartbeat 的周期性巡检不同，Cron 更适合有明确时间要求的定时工作。

OpenClaw 通过 CLI 命令管理定时任务。按当前官方口径，Cron 任务默认保存在 `~/.openclaw/cron/jobs.json`，更推荐用 `openclaw cron add`、`openclaw cron edit` 这类命令管理，而不是直接手改配置文件。

#### 方式一：自然语言配置（推荐新手使用）

直接在对话框中用自然语言下达指令：

    > 请设置一个定时任务：每天早上 9 点，自动抓取 Hacker News 前 10 条热帖，
    > 生成一份中文摘要，然后推送到我的飞书私聊

    [AI 自动解析并创建 Cron Job]
    定时任务已创建：
    - 名称：每日科技早报
    - 触发时间：每天 09:00 (UTC+8)
    - 执行内容：抓取 HN Top 10 → 中文摘要 → 飞书推送

#### 方式二：CLI 命令管理（精确控制）

使用 `openclaw cron` 系列命令来创建和管理定时任务：

```bash
# 运行指定的定时任务
$ openclaw cron run <job-id>

# 查看已配置的定时任务列表
$ openclaw cron list
```

**Cron 表达式速查表：**

| 表达式            | 含义         |
| -------------- | ---------- |
| `0 9 * * *`    | 每天早上 9:00  |
| `0 */2 * * *`  | 每 2 小时     |
| `*/30 * * * *` | 每 30 分钟    |
| `0 9 * * 1-5`  | 工作日早上 9:00 |
| `0 18 * * 5`   | 每周五下午 6:00 |
| `0 0 1 * *`    | 每月 1 号零点   |

> **进阶技巧：** 定时任务触发的 Agent 同样可以使用 `sessions_spawn` 工具调用来委派子任务。例如，早间简报任务可以先让技术专家检查 CI/CD 状态，再让战略顾问汇总行业动态，最后整合为简报。

### 2.3 引擎 C：Webhooks / Hooks（外部事件触发器）

**Webhooks** 让 Agent 能够对外部事件做出实时响应。当外部系统（如 GitHub、飞书）发生特定事件时，通过 HTTP 回调触发 Agent 执行相应操作。

OpenClaw 可以接收这类外部事件，但实际落地时通常还需要你自己的网关层或自动化平台做桥接。也就是说，这里更多是“实现思路”，不是只填一个字段就能直接跑起来的开箱即用功能。

#### 场景一：飞书多维表格数据变动触发

当团队成员在飞书多维表格中更新了客户状态，自动触发 Agent 跟进。

实现方式：在你的 API 网关层（如 n8n、自建中间件）接收飞书的事件回调，解析事件内容后，再把任务转发给对应的 Agent。

#### 场景二：GitHub 代码提交触发自动审查

当有人向你的仓库提交 PR 时，自动触发技术专家进行代码审查。

实现方式：配置 GitHub Webhook 指向你的中间件服务，中间件解析 PR 事件后，再把代码审查任务转发给技术专家 Agent。

#### 场景三：n8n 工作流集成

如果你使用 **n8n**（开源自动化工作流平台）来编排复杂的业务流程，可以将 OpenClaw 作为其中的一个智能节点。

一个典型的 n8n + OpenClaw 工作流示例：

    [n8n 工作流]
    触发器：每天 9:00 定时触发
      │
      ▼
    节点 1：HTTP Request → 调用你的 OpenClaw 接入层
      │
      ▼
    节点 2：等待 OpenClaw 返回结果
      │
      ▼
    节点 3：将结果格式化为 HTML 邮件
      │
      ▼
    节点 4：通过 SMTP 发送邮件给团队

> **注意：** 以上场景中的 Webhook 接收和路由逻辑通常需要你自行搭建一个 API 网关层（可以用 n8n、Cloudflare Workers 或简单的 Express 服务）来桥接外部事件和 OpenClaw。OpenClaw 本身负责 Agent 执行，事件接入层需要根据你的部署环境自行适配。

> **安全提醒：** 所有外部事件接入点都应配置签名验证。使用 HMAC-SHA256 等算法校验请求签名，防止未授权的外部请求触发你的 Agent。密钥应通过环境变量管理，不要硬编码在配置文件中。

## 3. 高阶架构安全：工具沙箱隔离（Tool Sandboxing）

当你把多个工具开放给自主运行的 Agent 时，安全就成了重中之重。OpenClaw 提供了 per-agent 的沙箱配置，确保工具调用在受控环境中执行。

### 3.1 沙箱配置

OpenClaw 的沙箱是在 `agents` 配置中按 Agent 粒度设置的。你可以在 `agents.defaults` 中设置全局默认值，也可以为单个 Agent 单独配置。

```json
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "anthropic/your-model-id"
      },
      "sandbox": {
        "mode": "off",
        "scope": "session"
      }
    },
    "list": [
      {
        "id": "personal-assistant",
        "name": "私人助理",
        "sandbox": {
          "mode": "off",
          "scope": "session"
        }
      },
      {
        "id": "tech-expert",
        "name": "技术专家",
        "sandbox": {
          "mode": "non-main",
          "scope": "agent"
        }
      },
      {
        "id": "community-mod",
        "name": "社区版主",
        "sandbox": {
          "mode": "all",
          "scope": "shared"
        }
      }
    ]
  }
}
```

**`mode` 可选值：**

| 值          | 说明                                  |
| ---------- | ----------------------------------- |
| `off`      | 不使用沙箱，工具直接在宿主环境执行                   |
| `non-main` | 非主会话（如 spawn 子会话、Webhook 触发的会话）使用沙箱 |
| `all`      | 所有会话都使用沙箱                           |

**`scope` 可选值：**

| 值         | 说明                      |
| --------- | ----------------------- |
| `session` | 每个会话独立一个沙箱实例，会话结束后销毁    |
| `agent`   | 同一个 Agent 的所有会话共享一个沙箱实例 |
| `shared`  | 所有 Agent 共享同一个沙箱实例      |

### 3.2 沙箱策略建议

| 场景                 | 推荐 mode            | 推荐 scope  | 理由                      |
| ------------------ | ------------------ | --------- | ----------------------- |
| 你亲自操作的私聊           | `off`              | -         | 你自己控制输入，风险可控            |
| 面向外部用户的 Agent      | `all`              | `session` | 每个用户会话隔离，防止互相影响         |
| Webhook/Cron 触发的任务 | `non-main` 或 `all` | `session` | 外部触发的任务应在沙箱中执行          |
| spawn 子会话          | `non-main`         | `agent`   | 子任务在沙箱中执行，同 Agent 可共享状态 |

**核心原则：** 哪怕 Agent 被诱导执行了 `rm -rf /` 之类的毁灭性命令，也仅仅是破坏了一个临时的沙箱实例，你的真实系统毫发无损。

![配图 7-4：沙箱 mode 和 scope 组合的可视化，展示不同配置下的隔离效果](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/7103a01fcd04440baf8a82b2ecf0bf40~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=928\&h=1152\&s=1364782\&e=png\&b=f2f1e3)

> **最佳实践：** 对于面向外部用户的 Agent（如社区版主、客服机器人），建议将 `mode` 设为 `all`、`scope` 设为 `session`，确保每个用户会话完全隔离。即使有人通过 Prompt 注入诱导 Agent 执行恶意操作，也无法影响其他会话或宿主系统。

## 4. 多智能体调试与监控

当你的多智能体系统运转起来后，你需要一套排查手段来确保一切正常。

### 4.1 状态查看与日志排查

OpenClaw 提供了基本的状态查看能力。你可以通过以下方式排查问题：

```bash
# 查看 OpenClaw 当前运行状态
$ openclaw status
```

对于更详细的日志，OpenClaw 的运行日志默认输出到标准输出。如果你使用 systemd 或 Docker 部署，可以通过对应的日志系统查看：

```bash
# systemd 部署时查看日志
$ journalctl -u openclaw -f

# Docker 部署时查看日志
$ docker logs -f openclaw
```

### 4.2 常见问题排查

| 问题现象          | 可能原因                 | 排查方法                                    |
| ------------- | -------------------- | --------------------------------------- |
| Agent 不响应     | 模型 API 超时或 Key 失效    | 检查运行日志中的 API 错误信息                       |
| 消息发到错误的 Agent | bindings 配置错误        | 检查 `bindings[]` 中 channel 和 agent 的对应关系 |
| 子会话创建失败       | 目标 Agent 未在 list 中定义 | 确认 `agents.list[]` 中包含目标 Agent 的 id     |
| 定时任务未执行       | Cron 服务未启动           | 使用 `openclaw cron list` 确认任务状态          |
| Agent 记忆混乱    | 多个 Agent 共享了相同的上下文   | 确认每个 Agent 的 id 唯一且配置独立                 |

### 4.3 成本控制建议

*   对于高频但低复杂度的任务（如社区问答），使用成本更低的模型（如 `deepseek/deepseek-chat`）
*   在 `agents.defaults` 中设置合理的默认模型，只为需要高能力的 Agent 单独指定昂贵模型
*   定期审查各 Agent 的使用频率，识别是否有异常循环调用并优化其 Prompt

## 小结

本章介绍了 OpenClaw 的多智能体架构和自动化引擎，让你的 AI 系统从单 Agent 升级为多 Agent 协同、7x24 小时自动运行的完整平台。

回顾本章的核心要点：

*   **多智能体配置**通过 `agents.list[]` 定义多个独立 Agent，通过 `bindings[]` 将渠道绑定到指定 Agent，实现人设和模型的隔离
*   **Session Tools**（`sessions_list`、`sessions_send`、`sessions_spawn`）是 Agent 的内置工具调用，让 Agent 之间能够感知状态、互相传话、委派任务
*   **三类触发方式**（Heartbeat、Cron、Webhooks/Hooks）让 Agent 不只会被动响应；其中 Cron 和 Heartbeat 是内建能力，Webhook/Hooks 通常还要接外部系统
*   **工具沙箱**通过 per-agent 的 `sandbox.mode` 和 `sandbox.scope` 配置，为多智能体架构提供了灵活的安全隔离

在下一章中，我们将运用前七章学到的所有知识，带你从零搭建一个真实的工作流——看看 OpenClaw 是如何包揽从选题挖掘、内容写作到全平台自动发布的自媒体全链路的。
