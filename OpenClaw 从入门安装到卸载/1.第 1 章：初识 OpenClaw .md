## 1. 为什么传统的 AI 聊天机器人不够用？

### 1.1 "一问一答"的天花板

日常使用的 **ChatGPT** 或 **Claude** 网页版，本质上是"被动响应式"的工具：你发一条指令，它回一段文字。关掉网页，它就立刻"失忆"。这种交互模式存在三个根本性的瓶颈：

| 痛点     | 具体表现                   | 影响           |
| ------ | ---------------------- | ------------ |
| 无记忆持久化 | 每次新对话都从零开始，无法记住你上周说过什么 | 重复沟通成本极高     |
| 无执行能力  | 只能生成文字，不能操作文件、浏览器或系统   | 输出无法直接落地     |
| 无自主性   | 必须你主动发起对话，它才会响应        | 无法 7x24 小时工作 |
| 无协作能力  | 单一模型单一对话，无法多角色并行       | 复杂任务效率低下     |

简单来说：传统 AI 只能生成文字，不能操作任何外部系统。你需要的是一个能直接执行任务的 AI 助手。

### 1.2 从"对话 AI"到"智能体 AI"的范式跃迁

业界正在经历一场从 **Chatbot**（聊天机器人）到 **Agent**（智能体）的范式跃迁。两者的本质区别在于：

*   **Chatbot** = 大语言模型 + 对话界面
*   **Agent** = 大语言模型 + 感知能力 + 执行能力 + 记忆系统 + 自主决策循环

**OpenClaw**（前身为 **ClawdBot** / **MoltBot**）正是这一方向的产物。它是一个开源的、可自托管的**个人 AI 助手系统**，运行在你的本地电脑或云服务器上，拥有完整的感知-思考-执行闭环。

它和普通对话 AI 的核心区别：

*   **能调用工具**：OpenClaw 可以打开浏览器提取网页数据、读写本地文件、执行 Shell 命令。它不只是生成文字，而是能直接操作你的系统。
*   **支持定时和后台任务**：OpenClaw 内置了 **Cron**（按固定时间运行任务）和 **Heartbeat**（按固定节奏做巡检或状态同步），可以把一些重复工作放到后台执行。
*   **长期记忆**：它通过工作区里的记忆文件保存长期信息。你之前确认过的偏好、约束和重要结论，后续可以再检索出来。

## 3. 典型使用场景

OpenClaw 的价值在于把 AI 从"只能聊天"变成"能执行任务"。以下是几个典型的应用方向。

### 3.1 多智能体协作

OpenClaw 支持配置多个 Agent。这里的 **Agent** 可以先理解成“分工不同的助手实例”；每个 Agent 都有自己的身份和职责。你可以在配置文件里定义多个 Agent，再通过 `bindings` 把不同的渠道、账号或会话路由到对应的 Agent。

> **注意**：多智能体的配置通过 `openclaw.json` 中的 `agents.list[]` 和 `bindings[]` 实现。这里的 `bindings` 可以理解成“把某类消息分配给哪个 Agent 的规则”。`AGENTS.md` 只是工作区提示文件，用来写角色说明，不是系统配置文件。详见第 7 章。

### 3.2 DevOps 自动化

这是开发者群体中常见的场景。前提是你已经接好了告警来源、Webhook 和对应工具权限。满足这些前提后，你可以让 OpenClaw 在收到线上报错时：

1.  接收错误通知
2.  分析错误日志，定位根因
3.  调用工具执行修复操作
4.  通过飞书 / Telegram 通知你审核结果

具体能做到什么程度，取决于你配置的工具权限和实际接入的服务。

### 3.3 内容生产辅助

如果你是内容创作者，也可以让多个 Agent 分工协作。前提是你已经把资料来源、写作工具和发布流程接进来：

*   **"研究员"Agent**：负责信息收集和素材整理
*   **"写手"Agent**：基于研究员的输出生成初稿
*   **"审校"Agent**：检查事实准确性和格式规范

这需要合理的 Agent 配置和工具权限设置，具体实现方式将在第 7、8 章详细介绍。

### 3.4 更多应用方向

| 场景   | 描述                | 涉及的工具/能力        |
| ---- | ----------------- | --------------- |
| 智能客服 | 自动回复客户咨询，复杂问题升级处理 | 浏览器工具、记忆检索、通讯渠道 |
| 学术助手 | 追踪论文、生成文献综述       | 浏览器工具、文件读写      |
| 数据监控 | 定时拉取数据，异常告警       | Cron 定时任务、浏览器工具 |
| 竞品监控 | 定期抓取竞品动态，生成分析报告   | 浏览器工具、文件读写      |

## 4. OpenClaw 的核心架构

OpenClaw 的核心由四大模块协同运作：**Gateway**（网关）、**Agent**（智能体引擎）、**Skills/Tools**（技能与工具）、**Memory**（记忆系统）。

![配图 1-1：OpenClaw 四大核心模块架构关系图](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/93e336794f954ed498461993ba8678a5~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1376\&h=768\&s=1266357\&e=png\&b=f8f4e7)

下面我们逐一拆解每个模块。

### 4.1 Gateway（网关）

**Gateway** 是 OpenClaw 的常驻服务进程。它负责：

*   **接收外部消息**：无论你从飞书、Telegram 还是 WhatsApp 发来指令，Gateway 统一接收和解析。
*   **路由和分发**：根据 `bindings` 配置，将消息路由到对应的 Agent 处理。`bindings` 就是“哪类消息交给哪个 Agent”的规则。
*   **鉴权和安全**：支持 `none`、`token`、`password`、`trusted-proxy` 等鉴权模式，默认绑定 `loopback`（仅本机访问），端口为 `18789`。
*   **暴露 Web 界面**：提供内置的 Web 聊天界面和 HTTP API 接口。

### 4.2 Agent（智能体引擎）

**Agent** 是真正执行任务的核心。每个 Agent 拥有独立的身份和行为模式，工作流程是一个 **感知-思考-执行** 循环：

1.  **感知**：接收来自 Gateway 路由过来的消息
2.  **思考**：调用大语言模型（如 Claude、GPT-4、DeepSeek 等）进行推理和规划
3.  **执行**：调用工具和技能完成具体操作（打开浏览器、写文件、发消息等）
4.  **反馈**：将执行结果返回，并决定是否需要继续下一步

Agent 的配置通过 `openclaw.json` 中的 `agents.list[]` 定义。此外，你可以在工作区中放置提示文件来补充 Agent 的行为说明：

*   `AGENTS.md`：工作区提示文件，用于描述 Agent 的角色和协作规则（注意：这是提示文件，不是系统配置文件）
*   `SOUL.md`：工作区提示文件，用于定义 Agent 的沟通风格和行为边界

### 4.3 Tools 与 Skills（工具与技能）

OpenClaw 中的能力分为两类，需要区分清楚：

*   **内置工具（Tools）**：OpenClaw 自带的能力，如浏览器操作（`browser`）、文件读写、Shell 命令执行、记忆检索（`memory_search`、`memory_get`）等。这些不需要额外安装。
*   **技能（Skills）**：通过 ClawHub 技能市场安装的扩展能力。每个 Skill 以 `SKILL.md`（含 YAML 前置元数据）为核心描述文件。安装和管理使用 `clawhub` 命令。

常见的内置工具：

| 类别    | 工具名                          | 能力说明                         |
| ----- | ---------------------------- | ---------------------------- |
| 浏览器操作 | `browser`                    | 打开网页、提取数据、填写表单（内置工具，非 Skill） |
| 文件系统  | 文件读写工具                       | 读写本地文件、管理目录                  |
| 系统命令  | Shell 执行                     | 执行 Shell 脚本和系统命令             |
| 记忆检索  | `memory_search`、`memory_get` | 搜索和获取历史记忆内容                  |

> 如果内置工具不能满足需求，可以通过 `clawhub install <skill-slug>` 从 ClawHub 安装社区技能。自定义开发技能的方式将在第 5 章详细介绍。

### 4.4 Memory（记忆系统）

**Memory** 负责存储和检索历史信息，让 Agent 具备长期记忆能力。

OpenClaw 的记忆系统基于工作区中的**记忆文件**：

*   `MEMORY.md`：主记忆文件，存储稳定、长期有效的信息
*   `memory/YYYY-MM-DD.md`：按日期组织的记忆文件，用来记录每天新增的重要内容
*   OpenClaw 会在后台为这些文件建立索引，方便后续检索

Agent 通过 `memory_search` 和 `memory_get` 工具来检索这些记忆。能不能顺利找回信息，取决于你记下来的内容是否清楚、索引是否已经建立。

> 长期记忆的事实来源是工作区里的记忆文件。它不是"永久记住一切"的魔法，而是一个需要维护的知识检索系统。

## 5. 五分钟快速体验

在正式进入部署教程之前，我们用一个简单的场景，让你对 OpenClaw 的工作方式建立直观感受。

假设你已经部署好了 OpenClaw（下一章会详细教你），并通过 Feishu channel 接入了飞书。现在你在飞书里对 Agent 说：

    @小爪 帮我调研一下 2025 年国内 AI Agent 赛道的融资情况，
    整理成表格，包含公司名、融资轮次、金额和主要产品，
    然后保存到我的飞书云文档里。

接下来会发生什么？

1.  **Gateway 接收消息**：飞书消息通过 Feishu channel 到达 Gateway，Gateway 再根据 `bindings` 规则把消息交给对应的 Agent。
2.  **Agent 开始思考**：Agent 调用大模型进行任务拆解——决定先用浏览器工具搜索信息，再提取详细数据。
3.  **工具执行操作**：Agent 依次调用搜索、浏览器等工具，收集融资记录。
4.  **Memory 存储上下文**：关键信息被写入记忆文件，后续对话中可以检索。
5.  **输出结果**：Agent 将整理好的表格回复到飞书对话中。能否直接写入飞书云文档，取决于你实际授予的权限和配置。

整个过程你只需要发一条消息，剩下的由 OpenClaw 自动完成。
