# 第 10 章：性能调优与安全防护

> 当你的 OpenClaw 从一个偶尔聊天的工具，升级为 7x24 小时处理文件、执行自动化任务、甚至对外提供服务的系统时，你会面临两个关键挑战：**响应性能**与**安全防护**。这一章会把 OpenClaw 相关配置和通用 Linux 运维做法分开讲，重点放在当前版本仍然适用的配置原则上。

## 本章学习目标

读完本章，你将能够：

*   了解什么时候适合开启 BBR，减少跨地域访问抖动
*   合理配置 Docker 容器的内存与资源限制，防止 OOM 宕机
*   配置 Nginx 或 Caddy 反向代理的完整安全方案（含 SSL/TLS）
*   在 `openclaw.json` 中正确设置网关认证，杜绝未授权访问
*   配置防火墙规则，仅暴露必要端口
*   搭建日志监控与告警体系
*   使用压力测试工具验证系统承载能力
*   建立一套完整的安全检查清单（Checklist）

## 1. 性能优化：降低响应延迟

在使用初期，你可能觉得 AI 回复挺快。但随着对话历史越来越长（比如连续对话超过 1000 轮），或者有大量并发请求涌入时，系统的响应延迟会飙升至 3-5 秒，甚至因 CPU 拉满而直接宕机。

要实现低成本的高并发，你需要从网络层、系统层、应用层分别进行优化。

### 1.1 网络层：开启 BBR 拥塞控制算法

如果你将 OpenClaw 部署在云服务器上，可以评估是否开启 Google 的 BBR 拥塞控制算法。相比传统的 Cubic 算法，BBR 更擅长探测可用带宽，在跨地域、高丢包或高抖动网络里通常会更稳定。

**检查当前拥塞控制算法：**

```bash
sysctl net.ipv4.tcp_congestion_control
```

**临时开启 BBR（重启后失效）：**

```bash
sudo sysctl -w net.core.default_qdisc=fq
sudo sysctl -w net.ipv4.tcp_congestion_control=bbr
```

**永久开启 BBR（写入系统配置）：**

```bash
# 追加配置到 sysctl.conf
echo "net.core.default_qdisc=fq" | sudo tee -a /etc/sysctl.conf
echo "net.ipv4.tcp_congestion_control=bbr" | sudo tee -a /etc/sysctl.conf

# 使配置立即生效
sudo sysctl -p

# 验证是否生效
sysctl net.ipv4.tcp_congestion_control
# 输出应为：net.ipv4.tcp_congestion_control = bbr

lsmod | grep bbr
# 输出应包含：tcp_bbr
```

> **注意**：BBR 需要 Linux 内核版本 >= 4.9。目前主流云服务器（Ubuntu 20.04+、CentOS 8+）均已满足此要求。如果你使用的是较老的系统，需要先升级内核。

### 1.2 系统层：Docker 容器资源限制与内存防爆

如果你使用的是 2 核 4G 的入门级云实例，**务必将 Docker 容器的内存上限限制为 3GB**，预留 1GB 给系统核心进程。否则一旦 OpenClaw 在处理大文件或高并发时吃满内存，Linux 的 OOM Killer 会直接杀掉进程，导致服务彻底宕机。

**方式一：docker-compose.yml 配置（推荐）**

```yaml
version: "3.8"
services:
  openclaw:
    image: ghcr.io/openclaw/openclaw
    container_name: openclaw
    restart: unless-stopped
    ports:
      - "18789:18789"
    volumes:
      - ./data:/app/data
      - ./config:/app/config
    environment:
      - NODE_ENV=production
    deploy:
      resources:
        limits:
          memory: 3G        # 硬性内存上限
          cpus: "1.5"       # 限制使用 1.5 个 CPU 核心
        reservations:
          memory: 1G        # 最低保障内存
          cpus: "0.5"       # 最低保障 CPU
    # 容器内存超限时的行为：优先回收而非直接杀死
    mem_swappiness: 10
    logging:
      driver: "json-file"
      options:
        max-size: "50m"     # 单个日志文件最大 50MB
        max-file: "3"       # 最多保留 3 个日志文件
```

> **提示**：`deploy.resources` 在 Docker Compose v3 中需要配合 `docker stack deploy` 或开启 `docker compose --compatibility` 模式才能生效。如果你使用的是 Docker Compose v2，可以直接使用 `mem_limit` 和 `cpus` 字段。

**方式二：docker run 命令行方式**

```bash
docker run -d \
  --name openclaw \
  --restart unless-stopped \
  --memory=3g \
  --memory-swap=4g \
  --cpus=1.5 \
  --memory-reservation=1g \
  -p 18789:18789 \
  -v $(pwd)/data:/app/data \
  -v $(pwd)/config:/app/config \
  -e NODE_ENV=production \
  --log-opt max-size=50m \
  --log-opt max-file=3 \
  ghcr.io/openclaw/openclaw
```

**实时监控容器资源占用：**

```bash
# 查看容器实时资源使用情况
docker stats openclaw

# 输出示例：
# CONTAINER   CPU %   MEM USAGE / LIMIT   MEM %   NET I/O       BLOCK I/O
# openclaw    12.5%   1.2GiB / 3GiB       40.0%   15.3MB/8.1MB  2.1MB/512KB
```

### 1.3 应用层：减少无效上下文，缩短模型处理时间

长对话变慢，常见原因是每次都把大量历史上下文一并发给模型。上下文越长，模型处理越慢，费用也越高。

配合第 6 章提到的本地向量库（SQLite + Vector Embedding）和 SSD 云硬盘，系统可以通过高速 I/O 进行混合搜索（向量语义搜索 + BM25 关键词匹配），只提取与当前问题最相关的 3-5 条历史上下文发送给大模型，而不是把所有聊天记录一股脑塞进去。

**优化原理示意：**

    优化前（暴力全量）：
    用户提问 → 携带全部 500 轮历史（100K Tokens）→ 大模型处理 → 3-5 秒响应

    优化后（向量检索）：
    用户提问 → 向量化 → 在本地向量库中检索 Top 3 相关上下文（600 Tokens）→ 大模型处理 → 0.5-0.8 秒响应

这类做法的核心目标，不是追求一个固定的百分比，而是只把当前问题真正需要的上下文发给模型。能节省多少费用、能快多少，取决于你的会话长度、检索质量和模型本身的延迟。

> **实操建议**：确保你的云服务器使用的是 SSD 硬盘而非 HDD。向量检索涉及大量随机读取操作，SSD 的 IOPS 性能是 HDD 的 100 倍以上。腾讯云 Lighthouse 默认配备 SSD 系统盘，无需额外配置。

### 1.4 CDN 加速与国内直连优化

如果你使用的是 Web 控制台，可以考虑让 CDN 分发前端静态资源，减少首屏加载时间。

**CDN 配置步骤：**

1.  **选择 CDN 服务商**：国内推荐腾讯云 CDN 或阿里云 CDN，海外推荐 Cloudflare（免费计划即可）。
2.  **添加加速域名**：在 CDN 控制台添加你的域名（如 `ai.yourdomain.com`），回源地址填写你的服务器 IP 和端口。
3.  **配置缓存规则**：
    *   静态资源（JS/CSS/图片）：缓存 7 天
    *   API 接口（`/api/*`）：不缓存，直接回源
    *   WebSocket/SSE 连接：设置为长连接透传，不缓存
4.  **开启 HTTPS**：在 CDN 上配置 SSL 证书（可使用 Let's Encrypt 免费证书）。
5.  **修改 DNS 解析**：将域名的 CNAME 记录指向 CDN 提供的加速域名。

对于国内用户，选择优质的 BGP 网络节点（延迟压在 20ms 以内）是保证"秒回"的物理基础。

### 1.5 性能监控与压力测试

部署完成后，你需要一套监控手段来持续观察系统健康状况，并在上线前进行压力测试。

```bash

# 查看磁盘 I/O 情况（向量检索依赖 SSD 性能）

```

**先做基础健康检查：**

```bash
# 先确认当前服务状态正常

```

**如果你要做 HTTP 压测，请压测你自己确认可访问的入口，不要默认写死 `/healthz`。下面以反向代理后的首页为例：**

OpenClaw 具有极高的系统执行权限——它能读写文件、执行 Shell 命令、访问网络。一旦被攻破，黑客不仅能控制你的 AI，还能直接控制你的电脑或服务器。请务必对照以下内容逐一排查你的系统。

### 2.1 前提一：修补反向代理的未授权访问漏洞

这是新手部署中最常见的安全疏漏。为了方便在外网用域名访问 Web 面板，很多人会用 Nginx 或 Caddy 做反向代理。但 OpenClaw 早期在设计上对"本地连接（127.0.0.1）"默认自动放行。

**风险**：当流量经过反向代理转发后，OpenClaw 会误以为所有外部请求都来自本地 127.0.0.1，从而直接给予最高权限的免密访问。攻击者可以直接打开你的控制面板，查阅聊天记录，甚至执行任意 Bash 系统命令。

**缓解措施**：必须在代理配置中正确传递客户端真实 IP 和安全头，同时在 `openclaw.json` 中强制启用网关认证并将 `bind` 设为 `loopback`。

#### Nginx 完整安全配置

```nginx
# /etc/nginx/nginx.conf 或 conf.d/openclaw.conf

http {
    limit_req_zone $binary_remote_addr zone=openclaw_limit:10m rate=10r/s;

    server {
        listen 80;
        server_name ai.yourdomain.com;
        return 301 https://$server_name$request_uri;
    }

    server {
        listen 443 ssl http2;
        server_name ai.yourdomain.com;

        ssl_certificate     /etc/letsencrypt/live/ai.yourdomain.com/fullchain.pem;
        ssl_certificate_key /etc/letsencrypt/live/ai.yourdomain.com/privkey.pem;

        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
        ssl_prefer_server_ciphers off;
        ssl_session_cache shared:SSL:10m;
        ssl_session_timeout 1d;
        ssl_session_tickets off;

        ssl_stapling on;
        ssl_stapling_verify on;
        resolver 8.8.8.8 8.8.4.4 valid=300s;

        add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
        add_header X-Content-Type-Options "nosniff" always;
        add_header X-Frame-Options "SAMEORIGIN" always;
        add_header X-XSS-Protection "1; mode=block" always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;

        location / {
            limit_req zone=openclaw_limit burst=20 nodelay;

            proxy_pass http://127.0.0.1:18789;
            proxy_http_version 1.1;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
            proxy_buffering off;
            proxy_cache off;
            proxy_read_timeout 86400s;
            proxy_send_timeout 86400s;
        }

        location ~ /\.(git|env|config) {
            deny all;
            return 404;
        }

        access_log /var/log/nginx/openclaw_access.log;
        error_log  /var/log/nginx/openclaw_error.log warn;
    }
}
```

#### Caddy 替代方案（自动 HTTPS，零配置证书）

如果你觉得 Nginx 配置太繁琐，Caddy 是一个优秀的替代方案。它最大的优势是**自动申请和续期 Let's Encrypt 证书**，无需手动配置 SSL。

```caddyfile
# /etc/caddy/Caddyfile

ai.yourdomain.com {
    # Caddy 自动处理 HTTPS 证书申请与续期

    # 反向代理到 OpenClaw
    reverse_proxy 127.0.0.1:18789 {
        # WebSocket 支持（自动检测，无需额外配置）

        # 传递真实客户端 IP
        header_up X-Real-IP {remote_host}
        header_up X-Forwarded-For {remote_host}
        header_up X-Forwarded-Proto {scheme}

        # 长连接超时（SSE 支持）
        transport http {
            read_timeout 0
        }

        # 如果你确认当前部署版本提供了稳定的健康检查接口，再单独添加主动健康检查
    }

    # 安全响应头
    header {
        Strict-Transport-Security "max-age=63072000; includeSubDomains; preload"
        X-Content-Type-Options "nosniff"
        X-Frame-Options "SAMEORIGIN"
        X-XSS-Protection "1; mode=block"
        Referrer-Policy "strict-origin-when-cross-origin"
        -Server
    }

    # 速率限制
    rate_limit {
        zone dynamic_zone {
            key {remote_host}
            events 10
            window 1s
        }
    }

    # 禁止访问敏感文件
    @sensitive path_regexp \.(git|env|config)
    respond @sensitive 404

    # 日志
    log {
        output file /var/log/caddy/openclaw.log {
            roll_size 50mb
            roll_keep 3
        }
    }
}
```

#### SSL/TLS 证书配置（Let's Encrypt）

如果你选择 Nginx，需要手动配置 Let's Encrypt 证书。使用 Certbot 工具可以一键完成：

```bash
# 安装 Certbot（Ubuntu/Debian）
sudo apt install certbot python3-certbot-nginx -y

# 自动申请证书并配置 Nginx
sudo certbot --nginx -d ai.yourdomain.com

# 验证自动续期是否正常
sudo certbot renew --dry-run

# 设置自动续期定时任务（通常 Certbot 安装时已自动配置）
sudo systemctl enable certbot.timer
sudo systemctl start certbot.timer
```

> **提示**：如果你使用 Caddy，则完全不需要手动配置证书——Caddy 会自动处理一切。这也是很多人选择 Caddy 的核心原因。

#### openclaw\.json 安全配置

无论使用哪种反向代理，都**必须**在 `openclaw.json` 中强制启用网关认证：

```json
{
  "gateway": {
    "port": 18789,
    "bind": "loopback",
    "auth": {
      "mode": "token",
      "token": ""
    }
  },
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "non-main",
        "scope": "agent"
      }
    }
  }
}
```

> **说明**：`gateway.bind` 设为 `"loopback"` 表示网关仅监听本机，外部无法直连，必须经过反向代理。`gateway.auth.mode` 支持四种模式：`none`、`token`、`password`、`trusted-proxy`。生产环境通常应选择 `token` 或 `password`。沙箱的当前官方口径是 `agents.defaults.sandbox.mode: off | non-main | all`，以及 `scope: session | agent | shared`。

**生成安全的认证 Token：**

```bash
# 方法一：使用 openssl 生成随机 Token
openssl rand -hex 32
# 输出示例：a3f8b2c1d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0

# 方法二：使用 Node.js 生成
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# 将生成的 Token 填入 openclaw.json 的 gateway.auth.token 字段
```

### 2.2 前提二：警惕 Skills 供应链投毒

ClawHub 技能市场虽然繁荣，但任何人只需一个 GitHub 账号就能上传插件。由于缺乏严格的审核机制，恶意 Skill 混入市场的风险不容忽视。

**攻击手段**：黑客会将恶意指令进行 Base64 编码，当你一键安装该技能后，它会在后台解码并利用 `curl` 命令从远程服务器下载攻击脚本直接执行。更隐蔽的手段包括在技能的 `postinstall` 钩子中植入后门，或者通过提示词注入让 AI 主动执行恶意命令。

![配图 10-4：Skills 供应链投毒攻击的典型攻击链路图，展示从恶意 Skill 上传到用户安装再到数据窃取的完整攻击流程](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/519154ccfc334072b4a16ebec8dd65b6~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1376\&h=768\&s=1450108\&e=png\&b=f5f3e6)

**缓解措施**：

1.  **手动审查 Skill 源码**：安装前务必阅读 Skill 的 `tools/main.js`（或入口文件），检查是否存在 Base64 编码字符串、`eval()`、`curl | bash` 等可疑模式。
2.  **隔离环境测试**：对于来路不明的技能，先在隔离环境（虚拟机或纯净 Docker 沙箱）中测试，观察其网络请求和文件操作行为。
3.  **API Key 安全管理**：API Key 必须使用环境变量管理，绝不能明文写在 `TOOLS.md` 或任何配置文件里。
4.  **定期审计已安装技能**：检查工作区中已安装的 Skill 目录，移除不再使用或来源不明的技能。

```bash
# 检查 Skill 目录中是否存在可疑的编码字符串
grep -r "base64\|eval\|exec\|curl.*|.*bash" ~/.openclaw/skills/

# 查看某个 Skill 的入口文件
cat ~/.openclaw/skills/<skill-name>/tools/main.js
```

### 2.3 前提三：开启工具沙箱隔离（Tool Sandboxing）

不要让 AI 在你的宿主机（真实系统）上无隔离地执行命令。这一点非常重要——OpenClaw 的核心能力之一就是执行 Shell 命令和操作文件系统，这意味着如果 AI 被恶意提示词注入（Prompt Injection）攻击，它可能会在你的真实系统上执行任意命令。

OpenClaw 提供了严格的会话安全边界设计：

*   **主会话（Main Session）**：你自己亲自操作的私聊拥有最高权限，可以直接在主机上执行命令。这是合理的，因为你本人就是系统的管理员。
*   **群聊与外部触发会话（Group/DM）**：由于多人参与风险极高（可能遭遇恶意用户的间接提示词注入攻击），这些会话在执行浏览器操作或 Shell 命令时，**必须默认在 Docker 容器沙箱中执行**。哪怕 AI 出了错或被黑客诱导执行了 `rm -rf /` 之类的毁灭性命令，也仅仅是破坏了一个临时的沙箱容器，你的真实物理机毫发无损。

**沙箱隔离的工作原理**：OpenClaw 会根据你配置的沙箱模式，决定哪些会话需要进入隔离环境执行工具。你需要关注的是当前官方支持的配置项，而不是自己再额外拼一套旧字段。

**确认沙箱配置已生效：**

```bash
# 检查 openclaw.json 中的沙箱配置
cat ~/.openclaw/openclaw.json | grep -A 5 '"sandbox"'

# 预期输出应包含：
# "sandbox": {
#   "mode": "non-main",
#   "scope": "agent"
# }

# 测试沙箱是否正常工作（在群聊中发送测试命令）
# AI 执行的命令应该在容器内运行，而非宿主机
```

## 3. 防火墙规则配置

防火墙是你的第一道防线。原则很简单：**只开放必要的端口，拒绝一切不必要的入站流量**。

### 3.1 使用 ufw（推荐，Ubuntu 系统）

```bash
# 启用 ufw
sudo ufw enable

# 默认策略：拒绝所有入站，允许所有出站
sudo ufw default deny incoming
sudo ufw default allow outgoing

# 允许 SSH（重要！否则你会被锁在外面）
sudo ufw allow 22/tcp comment 'SSH'

# 允许 HTTP 和 HTTPS（反向代理使用）
sudo ufw allow 80/tcp comment 'HTTP'
sudo ufw allow 443/tcp comment 'HTTPS'

# 注意：不要直接暴露 Gateway 端口（默认 18789）到公网！
# 优先使用 loopback 绑定 + 反向代理的方式访问
# sudo ufw allow 18789/tcp  # 错误！不要这样做

# 查看当前规则
sudo ufw status verbose

# 输出示例：
# Status: active
# To                         Action      From
# --                         ------      ----
# 22/tcp                     ALLOW       Anywhere    # SSH
# 80/tcp                     ALLOW       Anywhere    # HTTP
# 443/tcp                    ALLOW       Anywhere    # HTTPS
```

### 3.2 使用 iptables（CentOS/高级用户）

```bash
# 允许已建立的连接
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 允许本地回环
sudo iptables -A INPUT -i lo -j ACCEPT

# 允许 SSH
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# 允许 HTTP/HTTPS
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# 拒绝其他所有入站流量
sudo iptables -A INPUT -j DROP

# 保存规则（CentOS）
sudo iptables-save > /etc/sysconfig/iptables

# 保存规则（Ubuntu，需安装 iptables-persistent）
sudo apt install iptables-persistent -y
sudo netfilter-persistent save
```

## 4. 日志监控与告警配置

安全不是一次性的工作，你需要持续监控系统的运行状态，并在异常发生时第一时间收到告警。

### 4.1 OpenClaw 日志配置

OpenClaw 默认将日志输出到标准输出（stdout），你可以通过 Docker 的日志驱动将其持久化：

```bash
# 查看 OpenClaw 实时日志
docker logs -f --tail 100 openclaw

# 查看最近的错误日志
docker logs openclaw 2>&1 | grep -i "error\|warn\|fatal" | tail -50

# 导出日志到文件（用于分析）
docker logs openclaw > /var/log/openclaw/app.log 2>&1
```

### 4.2 Nginx 访问日志分析

```bash
# 查看最近的异常请求（状态码 4xx/5xx）
awk '$9 >= 400' /var/log/nginx/openclaw_access.log | tail -20

# 统计访问频率最高的 IP（排查暴力破解）
awk '{print $1}' /var/log/nginx/openclaw_access.log | sort | uniq -c | sort -rn | head -10

# 查看是否有人尝试访问敏感路径
grep -E "\.(git|env|config|bak|sql)" /var/log/nginx/openclaw_access.log
```

### 4.3 简易告警脚本

创建一个简单的监控脚本，当检测到异常时通过飞书 Webhook 发送告警：

```bash
#!/bin/bash
# /opt/openclaw-monitor.sh

WEBHOOK_URL="https://open.feishu.cn/open-apis/bot/v2/hook/your-webhook-token"
CONTAINER_NAME="openclaw"

# 检查容器是否在运行
if ! docker ps --format '{{.Names}}' | grep -q "^${CONTAINER_NAME}$"; then
    curl -s -X POST "$WEBHOOK_URL" \
        -H "Content-Type: application/json" \
        -d "{\"msg_type\":\"text\",\"content\":{\"text\":\"[告警] OpenClaw 容器已停止运行！请立即检查。\"}}"
fi

# 检查内存使用率是否超过 85%
MEM_USAGE=$(docker stats --no-stream --format '{{.MemPerc}}' $CONTAINER_NAME | tr -d '%')
if (( $(echo "$MEM_USAGE > 85" | bc -l) )); then
    curl -s -X POST "$WEBHOOK_URL" \
        -H "Content-Type: application/json" \
        -d "{\"msg_type\":\"text\",\"content\":{\"text\":\"[告警] OpenClaw 内存使用率已达 ${MEM_USAGE}%，请注意！\"}}"
fi

# 检查最近 5 分钟是否有大量 4xx 错误
ERROR_COUNT=$(awk -v d="$(date -d '5 minutes ago' '+%d/%b/%Y:%H:%M')" '$4 > "["d && $9 >= 400' /var/log/nginx/openclaw_access.log 2>/dev/null | wc -l)
if [ "$ERROR_COUNT" -gt 50 ]; then
    curl -s -X POST "$WEBHOOK_URL" \
        -H "Content-Type: application/json" \
        -d "{\"msg_type\":\"text\",\"content\":{\"text\":\"[告警] 最近 5 分钟内检测到 ${ERROR_COUNT} 个异常请求，可能正在遭受攻击！\"}}"
fi
```

```bash
# 添加定时任务，每 5 分钟执行一次
chmod +x /opt/openclaw-monitor.sh
(crontab -l 2>/dev/null; echo "*/5 * * * * /opt/openclaw-monitor.sh") | crontab -
```

## 小结

安全与性能是生产环境部署的两个基本前提。通过本章的系统级优化——从网络层的 BBR 算法、系统层的 Docker 资源限制、应用层的向量检索降本，到安全层的反向代理加固、防火墙配置、沙箱隔离和日志监控——你的 OpenClaw 现在具备了更好的性能表现和更完善的安全防护。

三个核心前提：**不要直接暴露 Gateway 端口到公网，优先使用 loopback 绑定 + 反向代理**、**不要盲目信任第三方技能，安装前审查源码**、**不要让 AI 在宿主机上无隔离地执行命令**。做到这三点，你的 OpenClaw 就能安全、稳定地长期运行。
