# 第 3 章：基础配置与接入第三方大模型

## 前言

在上一章中，我们已经把 OpenClaw 跑起来了，也通过 Onboarding 做了最基础的初始化。但要想让它稳定可用，你还需要把配置文件、工作区规则和模型接入方式真正理顺。

本章我们将完成三件关键的事：

1.  **解读配置文件体系**：搞懂 `openclaw.json` 的核心配置结构
2.  **编写工作区提示文件**：通过 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`MEMORY.md` 定义 Agent 的角色、行为和长期信息
3.  **接入大模型**：为 Agent 接入 LLM，从高性价比国产模型到海外模型，低成本完成对接

OpenClaw 本身是一个开源的 Agent 框架，它不自带固定模型。它最终能做什么、做得稳不稳，很大程度上取决于你接入了什么模型、给了什么规则、配了什么权限。

## 学习目标

读完本章，你将掌握以下核心技能：

*   理解 `openclaw.json` 配置文件的完整结构与各字段含义
*   掌握 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`MEMORY.md` 这几类工作区文件的基本写法
*   能够看懂 **DeepSeek**、**Kimi / Moonshot**、**GLM / ZAI**、**通义千问** 等常见模型的接入方式
*   了解中转 API 的一站式聚合方案与 OAuth 对接 GPT 的流程
*   掌握 `gateway.auth.mode` 和 `agents.defaults.sandbox` 两个关键配置的正确设置方式
*   具备排查 2026.3 版本配置变更的能力

## 3.1 解密 OpenClaw 的大脑中枢：配置文件体系

OpenClaw 的配置体系分为两层：**全局配置文件**和**工作区提示文件**。前者控制系统行为，后者定义 Agent 的角色与规则。

### 3.1.1 全局配置文件：`openclaw.json`

核心配置文件默认存放在 `~/.openclaw/openclaw.json` 路径下，采用 **JSON5** 格式（支持写注释，手动编辑非常方便）。

以下是一份按当前官方口径整理的示例，覆盖你日常最常用的核心字段：

```json5
{
  // ==================== 网关配置（Gateway） ====================
  "gateway": {
    "port": 18789,                      // 网关端口号，默认 18789
    "bind": "loopback",                 // loopback | lan | tailnet | auto | custom
    "auth": {
      "mode": "none"                    // [必填] 认证模式：none | token | password | trusted-proxy
      // 使用 token 模式时：
      // "mode": "token",
      // "token": "your-secret-token-here"
      // 使用 password 模式时：
      // "mode": "password",
      // "username": "admin",
      // "password": "your-password"
    }
  },

  // ==================== 模型配置（Models） ====================
  "models": {
    "providers": {
      "openai": {
        "apiKey": "sk-xxx"
      }
    }
  },

  // ==================== Agent 配置 ====================
  "agents": {
    "defaults": {
      "workspace": "~/.openclaw/workspace",
      "model": {
        "primary": "openai/gpt-5"
      },
      "sandbox": {
        "mode": "off",                   // off | non-main | all
        "scope": "session"               // session | agent | shared
      }
    },
    "list": [
      {
        "id": "main",                   // Agent 唯一标识
        "name": "小爪"                   // Agent 显示名称
      }
    ]
  },

  // ==================== 日志配置（Logging） ====================
  "logging": {
    "level": "info",                    // 日志级别：debug | info | warn | error
    "file": "~/.openclaw/logs/app.log", // 日志文件路径
    "maxSize": "50MB",                  // 单个日志文件最大体积
    "maxFiles": 5                       // 保留的日志文件数量
  }
}
```

各配置项中，最关键的三个模块是 `gateway`（决定你能不能连上）、`models`（决定 AI 有多聪明）和 `agents`（决定 AI 的身份和行为）。接下来我们逐一深入。

### 3.1.2 工作区提示文件：为 Agent 提供角色说明和行为指引

OpenClaw 的角色定义和行为规则不是写死在代码里的，而是通过工作区（Workspace）中的几类 **Markdown 文件**来动态拼装的。

> **重要区分**：`AGENTS.md`、`SOUL.md`、`TOOLS.md`、`MEMORY.md` 都属于工作区文件，用来补充角色说明、行为规则、工具使用说明和长期信息。它们不是系统配置文件，不能替代 `openclaw.json`。系统级配置（如网关端口、模型接入、沙箱模式等）必须在 `openclaw.json` 中设置。

这套设计的好处在于：你可以为不同的使用场景创建不同的工作区，每个工作区里的 Agent 拥有完全不同的角色定义和能力边界。

#### `AGENTS.md`——Agent 角色定义

这是 Agent 的基线规则文件。你在这里定义 Agent 的身份、能力范围和行为约束，也可以配置多 Agent 的协作规则。以下是一份模板示例：

```markdown
# Agent 配置

## 基本身份
- 名称：小龙虾助手
- 角色：全能型个人 AI 助理
- 语言：默认使用中文回复，用户使用英文时切换为英文

## 核心能力
- 文件管理：可以读取、创建、编辑和整理本地文件
- 信息检索：可以搜索网页、抓取网页内容、总结文档
- 代码辅助：可以编写、审查和调试代码
- 日程管理：可以创建提醒、管理待办事项
- 数据分析：可以处理 Excel/CSV 文件，生成图表

## 行为准则
1. 执行任何文件删除操作前，必须先向用户确认
2. 不得访问或修改系统关键目录（如 /etc、/System）
3. 涉及金额超过 100 元的操作（如下单、转账），必须二次确认
4. 遇到不确定的问题，优先询问用户而非自行猜测
5. 所有输出内容不得包含用户的隐私信息（手机号、身份证号等）

## 多智能体路由规则（可选）
- 群聊场景：使用"严谨技术助理"人格，回复简洁专业
- 私聊场景：使用"贴心生活助理"人格，语气亲切随和
- 编程场景：使用"代码审查专家"人格，注重规范和最佳实践
```

#### `SOUL.md`——性格与灵魂

这个文件定义了 AI 的语气、人设和价值观。同样的能力配置，搭配不同的 `SOUL.md`，你的 AI 可以呈现出截然不同的"人格"。以下提供三种风格模板供参考：

**风格一：专业严谨型（适合工作场景）**

```markdown
# 灵魂设定

## 性格特征
你是一位经验丰富的高级技术顾问，说话简洁有力，注重效率。

## 语气风格
- 回复简洁明了，不说废话
- 使用专业术语时附带简短解释
- 给出建议时附带理由和依据
- 遇到模糊需求时，先列出可能的理解再确认

## 价值观
- 准确性优先于速度
- 安全性优先于便利性
- 用户隐私不可侵犯
```

**风格二：活泼幽默型（适合日常陪伴）**

```markdown
# 灵魂设定

## 性格特征
你是一个热情开朗的朋友，喜欢用轻松幽默的方式帮人解决问题。
偶尔会开个小玩笑，但关键时刻绝对靠谱。

## 语气风格
- 语气亲切自然，像朋友聊天
- 适当使用 emoji 增加趣味性
- 复杂问题用生活化的比喻来解释
- 完成任务后偶尔来一句俏皮话

## 价值观
- 让用户开心是第一要务
- 鼓励和正面反馈优先
- 遇到困难时保持乐观态度
```

**风格三：学术研究型（适合科研场景）**

```markdown
# 灵魂设定

## 性格特征
你是一位严谨的学术研究助理，擅长文献检索、论文分析和数据处理。

## 语气风格
- 使用学术化的表达方式
- 引用观点时注明来源
- 对不确定的结论明确标注"待验证"
- 数据分析结果附带置信度说明

## 价值观
- 学术诚信是底线
- 数据和证据驱动决策
- 承认知识边界，不编造信息
```

#### `TOOLS.md`——工具使用备注

这个文件专门用于补充“工具怎么用、有哪些约束”。它可以写使用建议和注意事项，但**不负责真正开启或关闭工具权限**。工具是否可用，还是由运行环境、配置和当前会话决定。

当你安装了新的 Skill（技能）后，可以在这里补充使用说明和注意事项：

```markdown
# 工具使用指南

## 浏览器工具（browser）
- 抓取网页时优先使用无头模式，减少资源消耗
- 遇到需要登录的页面，提示用户手动完成认证后再抓取
- 单次抓取内容超过 10000 字时，自动进行摘要处理

## 文件操作工具（file-manager）
- 批量操作前先执行 dry-run（预演），展示将要执行的操作列表
- 删除操作默认移入回收站而非永久删除
- 处理大文件（>100MB）时提前告知用户预计耗时

## 终端命令工具（shell）
- 禁止执行 rm -rf / 等危险命令
- 长时间运行的命令（>30秒）需要告知用户进度
- 安装软件包前先确认用户的包管理器（brew / apt / yum）

## 代码执行工具（code-runner）
- Python 脚本执行前检查依赖是否已安装
- 执行结果超过 500 行时自动折叠，只展示关键输出
- 涉及网络请求的代码需要用户确认目标地址
```

#### `MEMORY.md`——长期记忆文件

`MEMORY.md` 用来放稳定、长期有效的信息，例如你的偏好、固定规则、项目背景和长期约束。除此之外，工作区里还可以有 `memory/YYYY-MM-DD.md` 这类按日期记录的记忆文件，用来保存每天新增的重要信息。

> **小贴士**：写这些文件时尽量避免互相冲突。可以把它们理解成四种分工不同的说明书：`AGENTS.md` 管角色，`SOUL.md` 管语气，`TOOLS.md` 管工具说明，`MEMORY.md` 管长期信息。

## 3.2 第三方大模型（LLM）低成本接入指南

这一节只讲**当前官方支持的接入路径**，不固定写价格表。模型价格、免费额度、上下文长度变化很快，真正决定你怎么接的，通常是这三个问题：

1.  你是想走**官方 provider**，还是接**兼容 OpenAI 的自定义接口**
2.  你需要的是**通用对话**、**长文档处理**，还是**代码和推理**
3.  你是想自己填 API Key，还是希望用 **OAuth / Portal 登录** 这类官方授权方式

### 3.2.1 先看接入路线，不先背价格表

| 路线                          | 当前官方口径                                                                                         | 适合场景             |
| --------------------------- | ---------------------------------------------------------------------------------------------- | ---------------- |
| DeepSeek                    | 常见做法是按 OpenAI 兼容接口接入自定义 provider                                                               | 预算敏感、通用中文任务      |
| Moonshot / Kimi             | 官方 provider 为 `moonshot`，默认端点 `https://api.moonshot.ai/v1`，中国环境可用 `https://api.moonshot.cn/v1` | 长文档、长上下文任务       |
| GLM                         | 当前官方主口径是 `zai` provider，模型示例 `zai/glm-5`                                                       | 通用推理、代码与结构化输出    |
| Qwen                        | 当前官方主口径是 `qwen-portal-auth` 插件 + `openclaw models auth login`                                  | 想走官方 Portal 授权流程 |
| OpenAI / Anthropic / Google | 按各自官方 provider 接入                                                                              | 海外原生模型场景         |

> **建议**：先选“接入方式”和“适合的任务类型”，再去看价格页。这样更不容易被过时信息带偏。

### 3.2.2 方案 A：常见国产模型接入

如果你主要在中文环境里使用 OpenClaw，国产模型通常更容易接、速度也更稳。下面只保留当前还符合官方口径的接法。

#### DeepSeek 接入配置

**DeepSeek** 常见的接法是按“兼容 OpenAI 的自定义 provider”来配置。也就是说，OpenClaw 不一定把它当成一个内置官方 provider，但你可以按兼容接口把它接进来。

第一步，前往 [DeepSeek 开放平台](https://platform.deepseek.com) 注册账号并创建 API Key。

第二步，修改 `openclaw.json` 中的模型配置。下面这个写法的重点是两件事：一是告诉 OpenClaw 这是兼容 OpenAI 的接口；二是把默认模型写成当前官方的 `provider/model` 形式：

```json5
{
  "models": {
    "providers": {
      "deepseek": {
        "api": "openai-completions",
        "baseUrl": "https://api.deepseek.com/v1",
        "apiKey": "sk-your-deepseek-api-key",
        "models": [
          { "id": "deepseek-chat" },
          { "id": "deepseek-reasoner" }
        ]
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "deepseek/deepseek-chat"
        // "primary": "deepseek/deepseek-reasoner"
      }
    }
  }
}
```

第三步，保存配置后重新启动 OpenClaw 即可生效。

#### 月之暗面 Kimi 接入配置

**Kimi / Moonshot** 的官方口径更明确，OpenClaw 当前直接给出了 `moonshot` provider。默认端点是 `https://api.moonshot.ai/v1`；如果你在中国环境下使用，也可以换成 `https://api.moonshot.cn/v1`。

前往 [Kimi 开放平台](https://platform.moonshot.cn) 获取 API Key 后，配置如下：

```json5
{
  "models": {
    "providers": {
      "moonshot": {
        "api": "openai-completions",
        "baseUrl": "https://api.moonshot.ai/v1",
        "apiKey": "sk-your-kimi-api-key",
        "models": [
          { "id": "kimi-k2.5" }
        ]
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "moonshot/kimi-k2.5"
      }
    }
  }
}
```

> **说明**：如果你在国内网络环境里用 Moonshot，常见做法是把 `baseUrl` 改成 `https://api.moonshot.cn/v1`。其余结构不变。

#### 智谱 GLM 接入配置

**GLM** 这一段最容易写旧。当前官方主口径已经不是早期常见的 `zhipu` 写法，而是直接使用 `zai` provider。

按当前官方口径，配置示例如下：

```json5
{
  "models": {
    "providers": {
      "zai": {
        "apiKey": "your-zai-api-key"
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "zai/glm-5"
      }
    }
  }
}
```

#### 通义千问接入配置

**通义千问** 当前官方主口径不是“手填一个 DashScope API Key 就算标准接入”，而是走官方的 Portal 授权流程。

推荐步骤如下：

```bash
openclaw plugins enable qwen-portal-auth
openclaw models auth login --provider qwen-portal --set-default
```

完成授权后，OpenClaw 会按官方 Portal 路线使用 Qwen。当前官方默认端点是 `https://portal.qwen.ai/v1`。

### 3.2.3 方案 B：中转 API 服务（一站式聚合）

如果你不想在每个平台分别注册并配置密钥，也可以使用**中转 API（Relay API）**。它的本质是：你只对接一个上游，再由这个上游去转发到不同模型。

配置方式与兼容 OpenAI 的自定义 provider 类似：

```json5
{
  "models": {
    "providers": {
      "relay": {
        "api": "openai-completions",
        "baseUrl": "https://your-relay-api.com/v1",
        "apiKey": "sk-your-relay-api-key",
        "models": [
          { "id": "your-model-id" }
        ]
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "relay/your-model-id"
      }
    }
  }
}
```

它的优点主要是：

*   **一个 Key 管多个上游**：你不用在很多平台分别维护配置
*   **切换模型更省事**：改默认模型时，通常不需要重写整套配置
*   **适合统一结算**：对团队来说更容易管理

> **注意事项**：中转 API 属于第三方服务。你要自己评估稳定性、隐私政策和日志保留方式，不要把它当成 OpenClaw 官方服务。

### 3.2.4 方案 C：海外原生模型（API Key 直连）

如果你身处海外，或者你的工作场景本来就依赖海外模型，也可以直接使用官方 provider。这里给你看的是结构，不固定写死某个具体型号，避免过几周就过时。

#### OpenAI 接入配置

前往 [OpenAI Platform](https://platform.openai.com) 注册开发者账号，在 Settings > API Keys 中创建 API Key 后，配置如下：

```json5
{
  "models": {
    "providers": {
      "openai": {
        "apiKey": "sk-your-openai-api-key"
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "openai/your-model-id"
      }
    }
  }
}
```

> **说明**：OpenAI 既支持标准 API Key 路线，也支持 OpenAI / Codex 的授权方式。具体走哪条路，以你当前账号和官方 provider 页面为准。

#### Anthropic 接入配置

```json5
{
  "models": {
    "providers": {
      "anthropic": {
        "apiKey": "sk-ant-your-anthropic-key"
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "anthropic/your-model-id"
      }
    }
  }
}
```

#### Google / Gemini 接入配置

```json5
{
  "models": {
    "providers": {
      "google": {
        "apiKey": "your-google-ai-key",
        "baseUrl": "https://generativelanguage.googleapis.com/v1beta"
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "google/your-model-id"
      }
    }
  }
}
```

## 3.3 版本更新注意事项（2026.3 配置变更）

在使用第三方模型接入并启动服务时，部分用户会遇到配置报错。以下是 OpenClaw 在 2026 年 3 月版本更新中引入的配置变更，请检查你的 `openclaw.json`。

### 3.3.1 变更 1：Gateway 认证模式变为必填

**现象描述**：升级后执行 `openclaw start` 时报错：

    [ERROR] Gateway authentication mode is required.
    Please set "gateway.auth.mode" in your openclaw.json.

**原因分析**：这是一个 **Breaking Change**（破坏性更新）。在旧版本中，Gateway 默认允许无认证的本地连接。新版本为了防范未授权访问，**强制要求显式设置 `gateway.auth.mode`**。

**解决方法**：打开 `openclaw.json`，在 `gateway` 配置中明确添加认证方式。当前官方口径有四种模式：`none`、`token`、`password`、`trusted-proxy`。最常见的是下面两种：

**方式一：Token 认证（推荐，适合个人使用）**

```json5
{
  "gateway": {
    "bind": "lan",
    "port": 18789,
    "auth": {
      "mode": "token",
      "token": "my-secret-token-2026"   // 自定义一个足够复杂的 Token
    }
  }
}
```

**方式二：密码认证（适合多人共享）**

```json5
{
  "gateway": {
    "bind": "lan",
    "port": 18789,
    "auth": {
      "mode": "password",
      "username": "admin",
      "password": "your-strong-password"
    }
  }
}
```

如果只是本机自用，可以保留 `bind: "loopback"` 并使用 `mode: "none"`。如果你的 OpenClaw 放在反向代理后面，由代理层负责认证，再考虑 `trusted-proxy`。

> **安全提醒**：只要 `bind` 不是 `loopback`，就不要再用无认证配置。无论选择哪种方式，请确保 Token 或密码足够复杂。

### 3.3.2 变更 2：沙箱模式默认行为调整

**现象描述**：模型接入成功了，AI 能正常回复消息，但在执行文件操作或终端命令时行为受限。

**原因分析**：新版本调整了沙箱（sandbox）的默认行为。沙箱配置位于 `agents.defaults.sandbox`，用于控制 Agent 执行操作时的隔离级别。

**解决方法**：在 `openclaw.json` 中调整沙箱配置：

```json5
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "deepseek/deepseek-chat"
      },
      "sandbox": {
        "mode": "off",                  // off | non-main | all
        "scope": "session"              // session | agent | shared
      }
    }
  }
}
```

以下是三种沙箱模式的对比：

| 模式           | 值            | 行为               | 适用场景         |
| ------------ | ------------ | ---------------- | ------------ |
| **关闭**       | `"off"`      | 不使用沙箱隔离          | 本地个人使用、开发调试  |
| **非主 Agent** | `"non-main"` | 仅子 Agent 在沙箱中运行  | 多 Agent 协作场景 |
| **全部**       | `"all"`      | 所有 Agent 均在沙箱中运行 | 生产环境、安全敏感场景  |

> **建议**：如果你在本地个人使用，`"off"` 即可。如果部署在服务器上供多人使用，建议设置为 `"non-main"` 或 `"all"`，为 Agent 的操作加一层安全隔离。

## 小结

本章完成了 OpenClaw 的关键配置收口。核心要点回顾：

1.  `openclaw.json` 是系统的核心配置文件，其中 `gateway`、`models`、`agents` 三个模块最为关键
2.  `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`MEMORY.md` 都是工作区文件，用于补充角色、语气、工具说明和长期信息，不是系统配置文件
3.  接模型时先分清三种路线：官方 provider、兼容 OpenAI 接口、自带 Portal/OAuth 授权
4.  中转 API 提供了一站式聚合方案，一个 Key 即可调用多种模型
5.  升级到 2026.3 版本后，注意检查 `gateway.auth.mode` 和 `agents.defaults.sandbox` 配置

目前我们只能在终端或 Web 页面里和 AI 沟通，这显然不够方便。在下一章中，我们将把 OpenClaw 接入飞书、Telegram 等即时通讯工具，让它成为你随时可以对话的 AI 助手。
